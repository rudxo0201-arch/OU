# OU 버전 관리 & 문서 협업 UX 기획

> 웹하드처럼 익숙하게, Git처럼 강력하게. 사용자는 Git을 모른다.

---

## 핵심 원칙

```
1. 사용자 용어만 사용 (Git 용어 절대 금지)
   branch → 초안    commit → 저장 지점    merge → 반영
   conflict → 겹침   diff → 비교          revert → 되돌리기
   fork → 복사본     pull request → 제안   staging → 준비

2. 자동 저장 + 수동 지점
   모든 편집은 자동 저장 (이벤트 소싱)
   사용자가 원할 때 "저장 지점" 찍기 (= commit)
   저장 지점 없어도 모든 순간으로 되돌리기 가능

3. 파일 복사 영원히 불필요
   같은 DataNode에 시간축이 내장
   "v2_최종_진짜.docx" 같은 상황 원천 차단
```

## 개인 작업 플로우

### A. 일반 편집 (웹하드 느낌)

```
사용자 행동              OU 내부 동작
─────────────────────    ────────────────────────────
문서 열기                DocumentView 렌더링
텍스트 수정              → node_event 자동 기록 (every change)
                         → 상태바: "자동 저장됨 15:32"
"저장 지점 찍기" 클릭    → 이름 입력 모달: "시험 범위 정리 완료"
                         → version_snapshot 생성
                         → 버전 바에 ◉ 추가
한 시간 후 "되돌리기"    → 버전 목록에서 선택
                         → 스냅샷 복원 OR 이벤트 역재생
```

### B. 시간여행

```
버전 바의 ● 을 드래그하면:
  - 문서가 해당 시점 상태로 실시간 변경
  - "과거를 보는 중" 배지 표시
  - 편집 불가 (읽기 전용)
  - "이 시점으로 되돌리기" 또는 "현재로 돌아가기" 버튼

이것이 "보고서_v1_v2_최종.docx"를 영원히 없애는 기능
```

## 공동 작업 플로우

### C. 그룹 문서 편집

```
1. 원본 존재 (그룹의 공식 문서)
2. 편집자가 "내 초안 만들기" 클릭
   → 원본 기반으로 독립 편집 공간 생성 (= branch)
   → 원본에 영향 없음
3. 자유롭게 편집
   → 내 초안에서만 변경됨
   → 실시간으로 원본 대비 차이점 표시 (초록/빨강)
4. "원본에 제안하기" 클릭
   → 변경 내용 요약 자동 생성
   → 메시지 입력: "침구학 섹션 추가했습니다"
   → Owner/Editor에게 알림
5. 검토자가 제안 확인
   → 변경 내용 비교 화면
   → 댓글로 소통
   → 반영 / 수정 요청 / 거절
6. 겹침(conflict) 발생 시
   → 겹친 부분만 하이라이트
   → 선택지: A안 / B안 / 직접 작성
   → 해결 후 자동 반영
```

### D. 실시간 협업 (선택적)

```
같은 문서를 동시 편집 시:
  - 각자 커서 표시 (이름 + 색상)
  - 편집 중인 문장 잠금 (짧은 시간)
  - 저장은 각자의 초안에 독립 반영
  - 충돌 없음 (각자 초안이니까)

원본 직접 편집 모드 (Owner 전용):
  - Google Docs 스타일 실시간 동기화
  - 이건 Phase 2 (CRDT 필요)
```

## UI 컴포넌트 설계

### 버전 바 (VersionTimeline)

```
위치: DocumentView 상단, 캔버스 모드 토글 아래
크기: 높이 32px, 전체 너비

구성:
  ○ 일반 자동저장 지점 (작은 점, 호버 시 시각 표시)
  ◉ 수동 저장 지점 (큰 점, 이름 라벨)
  ● 현재 위치 (가장 큰 점, 드래그 가능)

인터랙션:
  - 아무 점 클릭 → 해당 시점으로 이동
  - ● 드래그 → 시간여행 (슬라이더)
  - ◉ 호버 → 저장 지점 이름 + 시각 툴팁
  - 우클릭 → "이름 변경" / "되돌리기" / "비교"

디자인: OU 흑백 원칙
  - 선: var(--mantine-color-gray-7)
  - 일반점: var(--mantine-color-gray-6)
  - 저장점: var(--mantine-color-gray-3)
  - 현재점: white
```

### 초안 탭 (DraftTabs)

```
위치: 그룹 문서 상단
구성: 세그먼트 컨트롤

  [원본]  [내 초안]  [김민준의 초안]  [제안 2건 🔴]

인터랙션:
  - "원본" = 읽기 전용 (Editor가 아닌 경우)
  - "내 초안" = 자유 편집, 원본 대비 차이 표시
  - 다른 사람 초안 = 읽기 전용
  - "제안 N건" = 반영 대기 중인 제안 목록
```

### 변경 표시 (ChangeMark)

```
초안 편집 중 원본 대비 변경사항 인라인 표시:

추가된 문장:   배경 rgba(255,255,255,0.05) + 좌측 흰 바
삭제된 문장:   취소선 + c="dimmed"
수정된 문장:   밑줄 + 호버 시 원본 텍스트 툴팁

이 표시는 sentences 레벨에서 동작
(OU는 이미 sentence 단위 구조화가 되어있으므로 자연스러움)
```

## 데이터 모델 (신규 테이블)

```sql
-- 모든 DataNode 변경을 기록하는 이벤트 테이블
CREATE TABLE node_events (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  node_id     UUID NOT NULL REFERENCES data_nodes(id) ON DELETE CASCADE,
  user_id     UUID NOT NULL REFERENCES profiles(id),
  event_type  TEXT NOT NULL,  -- 'created','updated','restored','deleted'
  diff        JSONB,          -- { field: { old, new } } 변경 내용
  snapshot    JSONB,          -- 전체 상태 스냅샷 (저장 지점용)
  created_at  TIMESTAMPTZ DEFAULT now()
);

-- 사용자가 수동으로 찍는 저장 지점
CREATE TABLE version_snapshots (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  node_id     UUID NOT NULL REFERENCES data_nodes(id) ON DELETE CASCADE,
  user_id     UUID NOT NULL REFERENCES profiles(id),
  name        TEXT NOT NULL,             -- "시험 범위 정리 완료"
  description TEXT,
  snapshot    JSONB NOT NULL,            -- 전체 DataNode + sections + sentences 상태
  parent_id   UUID REFERENCES version_snapshots(id),  -- 이전 저장 지점
  created_at  TIMESTAMPTZ DEFAULT now()
);

-- 그룹 문서의 초안 (= branch)
CREATE TABLE drafts (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  node_id     UUID NOT NULL REFERENCES data_nodes(id) ON DELETE CASCADE,
  user_id     UUID NOT NULL REFERENCES profiles(id),
  base_snapshot_id UUID REFERENCES version_snapshots(id),  -- 분기 시점
  name        TEXT DEFAULT '내 초안',
  status      TEXT DEFAULT 'editing',  -- 'editing', 'proposed', 'merged', 'abandoned'
  draft_data  JSONB NOT NULL,          -- 초안의 현재 상태
  created_at  TIMESTAMPTZ DEFAULT now(),
  updated_at  TIMESTAMPTZ DEFAULT now()
);

-- 제안 (= merge request)
CREATE TABLE proposals (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  draft_id    UUID NOT NULL REFERENCES drafts(id) ON DELETE CASCADE,
  node_id     UUID NOT NULL REFERENCES data_nodes(id) ON DELETE CASCADE,
  author_id   UUID NOT NULL REFERENCES profiles(id),
  title       TEXT NOT NULL,
  message     TEXT,
  status      TEXT DEFAULT 'open',  -- 'open', 'approved', 'rejected', 'merged'
  diff        JSONB NOT NULL,       -- 원본 대비 변경 내용
  conflicts   JSONB,                -- 겹침 정보 (있으면)
  reviewed_by UUID REFERENCES profiles(id),
  resolved_at TIMESTAMPTZ,
  created_at  TIMESTAMPTZ DEFAULT now()
);

-- 제안에 대한 댓글
CREATE TABLE proposal_comments (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  proposal_id UUID NOT NULL REFERENCES proposals(id) ON DELETE CASCADE,
  user_id     UUID NOT NULL REFERENCES profiles(id),
  body        TEXT NOT NULL,
  sentence_id UUID,  -- 특정 문장에 대한 인라인 댓글 (optional)
  created_at  TIMESTAMPTZ DEFAULT now()
);
```

## 구현 우선순위

### Phase 1: 개인 버전 관리 (MVP)
- node_events 테이블 + 자동 이벤트 기록
- version_snapshots 테이블 + "저장 지점 찍기" UI
- 버전 바 (VersionTimeline) 컴포넌트
- 되돌리기 기능
- 비교 화면 (나란히 diff)

### Phase 2: 그룹 초안 & 제안
- drafts 테이블 + 초안 생성/편집
- proposals 테이블 + 제안 생성/검토
- 초안 탭 (DraftTabs) UI
- 변경 표시 (ChangeMark) 인라인 렌더링
- 겹침 해결 UI

### Phase 3: 실시간 협업
- Supabase Realtime 구독
- 커서 공유 (presence)
- 문장 잠금 (optimistic locking)
- CRDT 기반 동시 편집 (장기)

## OU만의 차별점 (vs Google Docs / Notion / Git)

```
Google Docs:  실시간 협업 강점, 버전 관리 약함, 구조화 없음
Notion:       블록 편집 강점, 버전 관리 거의 없음, 협업 제한적
Git:          버전 관리 최강, 비개발자 사용 불가, 텍스트 전용
드롭박스:     파일 저장, 버전 = 파일 복사, 협업 없음

OU:
  - 자동 이벤트 소싱 (모든 순간 기록, 저장 버튼 불필요)
  - sentence 단위 diff (파일이 아니라 의미 단위로 비교)
  - 뷰 독립 (같은 버전 데이터를 어떤 뷰로든 렌더링)
  - 초안/제안 시스템 (Git의 branch/PR을 일반인 언어로)
  - 편집 행위 자체가 DataNode화 (편집 성향 분석 가능)
  - 겹침 해결이 문장 단위 (파일 단위가 아님)
```
