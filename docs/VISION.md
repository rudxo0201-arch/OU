## 서비스 비전

**"인생 전반에 걸쳐 항상 쓸 수밖에 없는 개인 데이터 우주"**

메인 카피: **Just talk.**
서브 카피: 일정도, 할 일도, 아이디어도. OU가 듣고, 기억하고, 보여줍니다.

### 문제 정의

사람들은 LLM을 많이 쓴다. 그런데 이 데이터가 LLM 좌측 패널에 대화 형식으로만 남는다:
```
1) 재탐색이 어렵다 → 예전에 뭔가 물어봤는데 어디 있는지 못 찾음
2) 나만의 데이터화가 어렵다 → 대화가 쌓여도 연결되지 않음
3) 노션/옵시디언에 저장해도 텍스트 더미일 뿐, 활용에 불리한 형태
```

OU는 대화 과정이 모두 데이터화되는 시스템이다.
일정앱, 습관앱, 과제앱, 노트앱을 모두 대체한다.
켜자마자 채팅창에 치기만 하면 된다.

### 기존 LLM vs OU

```
기존 LLM:
  세션 박스에 가둠 → 좌측 패널 시간순 나열
  나중에 못 찾음

OU:
  세션 없음. 연속 스트림으로 쌓임.
  "어제 AI와 데이터 온톨로지 대화했지?" 필터
  → 관련 DataNode 렌더링 → 뷰 저장 → 항상 최신
```

### OU LLM의 해자: 트리플화를 염두에 둔 생성

```
일반 LLM:  답변 생성 → 사후 트리플 추출 → 노이즈 많음
OU LLM:    트리플화 염두에 둔 답변 생성 → 바로 저장 → 정갈한 온톨로지

사용자 눈에는 자연스러운 문장.
하지만 처음부터 표준 서술어 11개 안에서 관계 표현.
데이터가 쌓일수록 격차가 벌어지는 자기강화 루프.
```

## 킬러 데모 시나리오

```
1) 비로그인 접속 → 채팅 바로 시작
2) "다음주 일요일 희민이 결혼식" 입력
   → 그래프에 캘린더 노드 생성 (흐릿하게)
   → NodeCreatedBadge: "💫 일정 노드가 생성됐어요"
3) "나 결혼식 언제였지?" 질문
   → 캘린더뷰 렌더링 — 날짜 하이라이트
4) 하단 SaveNudge 등장:
   "이 대화가 곧 사라져요. 저장하고 계속 쓰기 →"
5) [저장하기] → Google 원클릭 가입
6) 가입 완료 → "아까 하던 얘기 계속해요 😊"
```

확장 킬러 씬:
```
지출 기록    → 차트뷰 (이번달 식비 얼마?)
독서 메모    → 북뷰 (읽은 책들)
아이디어     → 마인드맵뷰
인간관계     → 관계 그래프뷰
부고문자 붙여넣기 → schedule + relation DataNode 자동 생성
청첩장 붙여넣기   → schedule DataNode (날짜/장소/인물)

기획자 유즈케이스 (이 대화가 바로 페인포인트):
  지금 이 기획 대화 → OU였다면 실시간으로 DataNode
  아이디어 나오는 족족 자동 분류 → idea / knowledge DataNode
  "오늘 기획 대화에서 나온 아이디어 보여줘" → 전부 뷰로 렌더링
  문서로 출력 → 기획서 자동 생성

  지금 이 대화처럼 Claude에게 "반영해줘" "기록해줘"
  계속 요청해야 하는 답답함 → OU가 해결

킬러 카피:
  "대화하는 족족 데이터가 됩니다"
  "아이디어는 사라지지 않아요"
  "지금 이 대화도 우주가 되고 있어요"
```

### SNS도 전부 DataNode + DataView다

```
페이스북이 하는 일:
  이름/직장/학교/관계 DataNode → "프로필뷰"로 렌더링

인스타그램이 하는 일:
  이미지 DataNode → "그리드뷰"로 렌더링

트위터가 하는 일:
  텍스트 DataNode → "피드뷰"로 렌더링

링크드인이 하는 일:
  경력/relation DataNode → "경력 프로필뷰"로 렌더링

→ 모두 DataNode + DataView
→ OU는 이미 이 구조로 설계되어 있다

기존 SNS:
  데이터가 플랫폼에 귀속 (내 데이터지만 페북 소유)
  플랫폼이 정해준 양식에 채워넣기
  뷰 선택권 없음

OU:
  데이터가 내 DB에 있음
  프로필 = 내 DataNode 중 공개 선택한 것들의 뷰
  "나를 어떻게 보여줄래?" → 내가 뷰 결정

  지식 그래프로 → 내가 뭘 알고 있는지
  관계 그래프로 → 내가 누구와 연결됐는지
  타임라인으로  → 내가 어떻게 성장했는지
  포트폴리오뷰로 → 내가 뭘 만들었는지
  같은 나, 다른 뷰. 맥락에 따라 다른 프로필.
```

### SNS 상호작용도 전부 DataNode

```
팔로우 = 구독 시스템 (이미 설계됨)
피드   = 구독한 사람들의 DataNode 시간순 렌더링 (이미 설계됨)
좋아요 = {subject: 나, predicate: likes, object: 그 DataNode}
댓글   = 새 DataNode + 원본 DataNode와 relation 연결
공유   = 그 DataView를 내 우주에 가져오기

→ 추가로 설계할 게 없음
→ 이미 있는 구조에 피드뷰 + 프로필뷰를 추가하면 됨
```

### OU SNS가 기존 SNS보다 강력한 이유

```
기존 SNS:
  텍스트/이미지/영상만 공유
  받은 사람은 그냥 "봄"
  데이터가 플랫폼에 귀속

OU SNS:
  DataView를 공유
  받은 사람은 "내 DB에 가져올 수 있음"
  → 누군가의 독서 기록을 내 우주에 연결
  → 누군가의 강의 DataNode를 내 그래프에 추가
  → 데이터가 살아있는 채로 이동

  언어 중립:
  한국인이 올린 지식 그래프
  → 일본인에게 일본어로 렌더링
  → 기존 SNS는 이게 불가능

락인:
  기존 SNS: 플랫폼이 데이터를 인질로 잡음
  OU:       사용자가 자발적으로 우주를 키움
            → 이탈하면 이 우주를 잃는 것
            → 훨씬 강력한 락인
```

### 회원 (Member)

```

```

## UNRESOLVED 엔티티 & 정확도 높이기

### 문제

```
"나 오늘 걔랑 싸웠어"
→ "걔" = 누구? → 맥락 없이 구조화 불가
```

### 해결: 2단계 저장

```
1단계: Raw 캡처 (무조건 저장)
  "걔" → UNRESOLVED 플래그 → 플레이스홀더 노드 (그래프에 흐릿하게)

2단계: 점진적 해소 (Progressive Resolution)
  대화가 쌓이면서 자동으로 해소
  "민준이가 또 그랬어" → "걔" = 민준 가능성↑ → 자동 병합
```

### 해소 방식 3가지 (동시 운영)

```
A) 자동 해소 (기다린다)
   맥락이 쌓이면 시스템이 백그라운드에서 추론
   해소 시 조용히 알림: "💡 '걔'가 민준님으로 연결됐어요"

B) 초반 안내 멘트
   "OU가 당신을 이해할수록 답변이 정교해져요!"
   부담 없이 더 많이 말하게 유도

C) "정확도 높이기" 탭 (/accuracy) — 받은 편지함 모델
   UNRESOLVED 리스트 + 컨텍스트 스니펫 + 객관식 선택지
   처리되면 자동 삭제 (맥락으로 자동 해소된 것도 조용히 삭제)
   왜 객관식: 주관식 입력 허용 시 스키마가 무너짐
              선택지로 표준 노드로 수렴시킴
```

### 스키마 가디언

```
"지피티" 입력 → 사용자 수준 파악 전
  precision 1 (GPT)로 임시 저장 (조용히)

userLevel = intermediate 파악 후:
  "어떤 의미로 쓰셨나요?
   1) GPT (ChatGPT)  2) LLM (언어모델)  3) AI  4) 직접 입력"
  → 선택 → 표준 노드 업데이트

원칙: 정확도보다 진입장벽 제거 먼저. 일단 저장, 점진적 정제.
```

## 회원가입 유도 & 온보딩

### 가입 유도

```
원칙: 가치 먼저 → 가입 나중
      "이미 이렇게 됐어요, 저장하려면 가입하세요" ✅

비로그인 5턴 → 훅 타이밍:
  타이밍 1: 5턴 후 "대화가 쌓이고 있어요. 사라지기 전에 저장하세요"
  타이밍 2: 의미있는 뷰 생성 시 "저장하시겠어요?"
  타이밍 3: 세션 종료 시 "X개 노드 생성됨. 저장 안 하면 사라져요"

가입: Google 원클릭 (최우선)
가입 후: 이전 대화 그대로 이어짐 ("아까 하던 얘기 계속해요 😊")
```

### 온보딩 (토스 스타일 대화형)

```
신규 가입 → OU가 먼저 말 걸기:
  OU: "안녕하세요! 뭐라고 불러드릴까요?"
  OU: "요즘 주로 어떤 것들 생각하고 계세요?"
  → 2~3턴 자연스러운 대화 → 일반 채팅으로 전환

자기개선 알고리즘:
  이탈 시점 추적 → A/B 테스트 → 최적 질문 수 탐색
  너무 짧으면 맥락 부족, 너무 길면 이탈
```

## OU의 청사진 — 인터넷 위의 공통 레이어

### 핵심 통찰

```
OU는 인터넷 위에 올라간 모든 서비스의
공통 레이어(DataNode + DataView)를 선점하는 것이다.

카톡, 인스타, 유튜브, 쿠팡이 각각 따로 만든 것들을
OU는 하나의 아키텍처로 다 담는다.
```

### 모든 서비스는 DataNode + DataView다

```
카카오톡:   채팅 DataNode → 채팅 피드뷰
카카오쇼핑: 상품 DataNode → 쇼핑뷰
카카오TV:  영상 DataNode → 방송뷰
카카오페이: finance DataNode → 결제뷰
카카오택시: schedule DataNode → 위치뷰

페이스북:   relation/schedule DataNode → 프로필뷰 + 타임라인뷰
인스타그램: 이미지 DataNode → 그리드뷰
트위터:     knowledge/idea DataNode → 피드뷰
링크드인:   relation/knowledge DataNode → 경력 프로필뷰
유튜브:     영상 DataNode → 영상 피드뷰

→ 이것들이 하는 일 전부가 이미 OU 아키텍처 안에 있다
```

### 채팅을 먹으면 → 카톡이 점유한 사업 분야 전체

```
카톡이 채팅 트래픽을 기반으로 붙인 것들:
  쇼핑 (카카오쇼핑)  → 상품 DataNode + 구매뷰
  방송 (카카오TV)    → 영상 DataNode + 방송뷰
  결제 (카카오페이)  → finance DataNode + 결제뷰
  택시 (카카오택시)  → schedule + 위치 DataNode + 지도뷰
  웹툰               → 콘텐츠 DataNode + 웹툰뷰
  게임               → 그래프뷰 게임 생태계 (이미 설계됨)
  선물하기           → DataView를 선물로 전송

OU가 같은 이유로 가능:
  사람들이 채팅창에 매일 접속 → 트래픽의 중심
  거기에 붙이면 됨

차이점:
  카톡: 기능을 억지로 붙임 (채팅앱인데 쇼핑? 이질감)
  OU:   모든 게 DataNode + DataView로 일관됨
        쇼핑도 DataNode, 방송도 DataNode
        → 이질감 없이 자연스럽게 흡수
```

### SNS를 먹으면 → SNS가 점유한 광고시장 전체

```
SNS 광고시장이 거대한 이유:
  사람들이 피드를 매일 봄 → 광고 노출
  인플루언서가 팔로워를 가짐 → 구독 기반 마케팅

OU SNS도 동일:
  피드뷰 → 인기 DataView에 광고 (이미 수익화에 설계됨)
  구독 시스템 → 인플루언서 DataNode 구독 (이미 설계됨)
  라이브 커머스 → 영상 DataNode + 실시간 구매뷰
  브랜드 계정  → 기업 DataNode + 브랜드뷰

추가로:
  기존 SNS 광고: 플랫폼이 사용자 데이터를 팔아서 타겟팅
  OU 광고: 사용자 DataNode 기반 컨텍스트 광고
           → 더 정확한 타겟팅 가능
           → 개인정보 직접 판매 없이 (데이터는 내 DB에)
```

### 왜 OU가 이걸 다 흡수할 수 있는가

```
카톡/SNS가 각 사업을 할 수 있었던 이유:
  사람들이 거기 모여 있어서
  채팅/피드가 트래픽의 중심이어서

OU도 같은 이유:
  매일 쓰는 채팅창 (LLM과 대화)
  매일 보는 피드 (DataView SNS)
  → 트래픽의 중심이 되면 파생 사업 자연스럽게 붙음

그런데 OU는 하나 더 있다:
  데이터가 내 DB에 쌓임
  쌓일수록 이탈 불가능
  → 기존 SNS/카톡보다 훨씬 강력한 락인

기존: 플랫폼이 데이터를 인질로 잡음
OU:   사용자가 자발적으로 우주를 키움
      이탈하면 이 우주를 잃는 것
```

### 프로필 = 나의 DataNode 공개 뷰

```
기존:
  프로필 = 플랫폼이 정해준 양식에 채워넣기
  링크드인용 따로, 인스타용 따로, 페북용 따로

OU:
  프로필 = 내 DataNode 중 공개 설정한 것들의 뷰
  내가 원하는 뷰 형식으로 렌더링

  "나를 어떻게 보여줄래?"
  → 지식 그래프뷰  : 내가 뭘 알고 있는지
  → 관계 그래프뷰  : 내가 누구와 연결됐는지
  → 타임라인뷰     : 내가 어떻게 성장했는지
  → 포트폴리오뷰   : 내가 뭘 만들었는지
  → 경력 프로필뷰  : 링크드인 대체

  같은 나, 다른 뷰. 맥락에 따라 다른 프로필.
  플랫폼마다 따로 관리할 필요 없음.
```

### 흡수 가능한 시장 전체 지도

```
1단계 (MVP): LLM 채팅 → 일정앱/노트앱/가계부/습관앱 대체
2단계:       그룹/구독 → 카톡 그룹/단톡방 대체
3단계:       SNS 피드/프로필 → 인스타/페북/트위터 대체
             + SNS 광고 시장 흡수
4단계:       채팅 커머스 → 카톡쇼핑/라이브커머스 흡수
5단계:       교육 시장 → 강의/문제지/오답노트 (이미 설계됨)
6단계:       방송/콘텐츠 → 유튜브/카카오TV 흡수
7단계:       결제/금융 → 카카오페이/토스 흡수
...단계 무한 확장

핵심: 단계마다 억지로 기능을 붙이는 게 아니라
      DataNode + DataView 아키텍처가 자연스럽게 확장됨
```

### 아키텍처가 이걸 막으면 안 된다 (설계 원칙)

```
지금 설계하는 코드가 나중 확장을 막으면 안 된다.

DataNode 스키마 확장 가능:
  상품 DataNode → domain: 'product' 추가하면 됨
  방송 DataNode → domain: 'broadcast' 추가하면 됨
  결제 DataNode → domain: 'finance' 이미 있음

뷰 레지스트리 플러그인 구조:
  쇼핑뷰 → 레지스트리에 등록만 하면 됨
  방송뷰 → 레지스트리에 등록만 하면 됨
  새 뷰 추가해도 기존 코드 수정 불필요

채팅/피드가 DataView 기반:
  → 이미 설계됨
  → 어떤 DataNode든 뷰로 만들어 전송 가능

지금은 1단계만 구현한다.
하지만 아키텍처는 7단계까지 담을 수 있어야 한다.
```

## DataView = 프론트엔드 (근본적 재정의)

```
컴퓨터 과학의 본질:
  백엔드 (DB)     = 데이터
  프론트엔드 (UI) = 데이터를 보여주는 것
  = DataNode + DataView (완전히 동일한 구조)

OU가 새로 발명한 게 아니라
웹의 본질 (DB + 프론트엔드) 을
개인이 소유하고, 커스텀하고, 거래하게 만드는 것.

차이점 딱 하나:
  기존: 프론트엔드가 서비스에 종속
        인스타 UI = 인스타 데이터만 보여줌

  OU:   DataView가 데이터에서 분리
        같은 DataNode → 어떤 DataView로든 렌더링
        DataView 자체를 사고팔 수 있음
        사용자가 직접 만들 수 있음

최종 정의:
  OU = 개인 DB + 그 DB의 프론트엔드를 무한으로 생성하는 플랫폼
```

## 챗봇뷰 = 나를 학습한 AI

```
챗봇도 하나의 뷰다.
내 DataNode로 RAG한 LLM = 나를 학습한 AI = 대화형 DataView

같은 DataNode 하나로:
  표로 보기 / 그래프로 보기 / PDF로 출력 / 챗봇으로 대화
  전자책으로 읽기 / 웹사이트로 공개 / 문제집으로 풀기

= 같은 데이터, 다른 프론트엔드
= OU의 본질
```
