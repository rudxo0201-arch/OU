# IDEAS.md — 아이디어 로그 (누락 방지용)

> 이 파일은 기획 대화에서 나온 모든 인사이트와 아이디어를 맥락과 함께 기록한다.
> 당장 구현 안 해도 된다. 아이디어가 사라지면 안 된다.
> 구현 완료된 것은 ✅ 표시. 미구현은 💡 표시.

---

## 핵심 인사이트 로그

### 💡 LLM 대화가 데이터로 쌓이지 않는 문제
```
사람들은 LLM을 많이 쓴다. 대화하면서 인사이트를 얻는다.
그런데 이 대화가 좌측 패널에 시간순으로만 쌓인다.
→ 재탐색 불가, 연결 불가, 데이터로 활용 불가

OU의 출발점:
  대화 과정이 모두 데이터화되는 시스템
  세션 없음. 연속 스트림. 필터 + 뷰로 꺼내 씀.
```

### 💡 트리플화를 염두에 둔 LLM 생성 (온톨로지 해자)
```
일반 LLM: 생성 → 사후 트리플 추출 → 노이즈
OU LLM:  처음부터 표준 서술어 11개 안에서 생성
         → 온톨로지가 처음부터 정갈함
         → 데이터 쌓을수록 경쟁사가 따라올 수 없음
```

### 💡 ou-study: YouTube → DataNode 파이프라인
```
YouTube URL → 트랜스크립트 → sections → sentences → triples
현재 독립 Next.js 툴로 존재 (ou-study/)
→ OU 메인과 통합 예정
→ 관리자 기본 DB 구축의 주요 수단
```

### ✅ 데이터뷰 = 필터 × 템플릿
```
같은 데이터를 어떤 형태로든 꺼내 쓴다.
DataView = Filter × Template → Render
저장된 뷰 = 살아있는 뷰 (항상 최신 데이터 반영)
```

### 💡 프로필도 데이터뷰다 (SNS 정복 가능)
```
페이스북 프로필:
  이름/직장/학교/관계 DataNode → "프로필뷰"로 렌더링
  → 이게 전부다. DataNode + DataView.

인스타: 이미지 DataNode → 그리드뷰
트위터: 텍스트 DataNode → 피드뷰
링크드인: 경력 DataNode → 경력 프로필뷰

→ 기존 SNS가 하는 일 전부 = OU 아키텍처 안에 있음

OU 프로필:
  내 DataNode 중 공개 설정한 것들의 뷰
  지식 그래프뷰 / 관계 그래프뷰 / 타임라인뷰 / 포트폴리오뷰
  같은 나, 다른 뷰. 플랫폼마다 따로 관리 불필요.
```

### 💡 채팅 먹으면 → 카톡 사업 전체 흡수
```
카톡이 채팅 기반으로 붙인 것들:
  쇼핑 / 방송 / 결제 / 택시 / 웹툰 / 게임 / 선물하기

OU도 같은 이유로 가능:
  사람들이 채팅창에 매일 접속 = 트래픽의 중심
  거기에 붙이면 됨

차이점:
  카톡: 채팅앱에 쇼핑 억지로 붙임 (이질감)
  OU:   모든 게 DataNode + DataView로 일관됨
        쇼핑 = 상품 DataNode + 구매뷰 (이질감 없음)
```

### 💡 SNS 먹으면 → 광고시장 전체 흡수
```
SNS 광고시장이 거대한 이유:
  피드를 매일 봄 → 광고 노출
  인플루언서 팔로워 → 구독 기반 마케팅

OU SNS도 동일:
  피드뷰 → 인기 DataView에 광고
  구독 시스템 → 크리에이터 수익화
  라이브 커머스 → 영상 DataNode + 실시간 구매뷰

추가 장점:
  기존 SNS 광고: 플랫폼이 사용자 데이터를 팔아서 타겟팅
  OU 광고: 사용자 DataNode 기반 컨텍스트 광고
           → 더 정확한 타겟팅 (개인정보 직접 판매 없이)
```

### 💡 SNS 상호작용 자체가 DataNode
```
팔로우 = 구독 (이미 설계됨)
피드   = 구독한 DataNode 시간순 렌더링 (이미 설계됨)
좋아요 = {subject: 나, predicate: likes, object: 그 DataNode}
댓글   = 새 DataNode + 원본 DataNode와 relation 연결
공유   = DataView를 내 우주에 가져오기

→ 추가 설계할 게 없음. 피드뷰 + 프로필뷰만 추가하면 됨.
```

### 💡 데이터뷰가 강의/강의안이 될 수 있다 (교육 시장)
```
강의 DataNode → 강의뷰 → 수강생 구독
강사가 수정 → 즉시 반영
수강생은 자신의 노트 DataNode 추가
→ 강의 + 내 노트 = 하나의 우주

언어 중립:
  한국어 강의 → 외국 수강생에게 영어로 렌더링
  → 글로벌 강의 플랫폼 가능
```

### 💡 그래프뷰가 강의 콘텐츠 생성기다 (강사 플라이휠)
```
강사가 DataNode를 쌓으면
그래프뷰가 강사도 몰랐던 연결을 보여준다
→ 새 강의 주제 무한 생성
→ 더 많이 쌓고 싶어짐 → 플라이휠

예: 한의사가 황기/인삼 DataNode 쌓음
    → "보기" 클러스터 → "보기약 비교 강의" 주제 발견
    → "폐" 중심 엣지 → "폐 관련 본초 총정리" 발견
```

### 💡 문제지 시장도 다 흡수 가능
```
현재 수학 문제지 시장 유행:
  단원/난이도 선택 → PDF 자동 생성 → 끝

OU:
  문제 DataNode → 어떤 뷰로든:
    PDF뷰: 문제지
    퀴즈뷰: 인터랙티브 풀기
    오답 DataNode: 자동 생성
    오답 PDF뷰: 오답노트
    취약점 그래프뷰: "내가 틀리는 유형"
    맞춤 문제지: 취약 단원만 추출 → PDF

데이터는 하나, 뷰는 무한.
풀수록 데이터 쌓임 → 더 정교한 맞춤 문제지.
```

### 💡 원본 파일 보존 필수 (양방향 연결)
```
PDF A → DataNode 생성
→ 원본 PDF A와 연결되어야 함
→ DataNode에서 "원본 보기" 가능
→ 해당 페이지/섹션으로 자동 스크롤

파싱 실패해도 원본은 R2에 보존 (재시도 가능)
원본 손실은 복구 불가 → 원본 보존이 절대 최우선

모든 파일 형식이 OU 안에서 렌더링될 수 있어야 한다.
→ .ou 시대의 전제 조건
```

### 💡 데이터가 쌓이면 뷰를 추천해줘야 한다
```
사용자는 뷰를 몰라도 된다.
데이터가 쌓이면 OU가 먼저 제안한다.

일정 5개 → "캘린더뷰 어떨까요?"
독서 여러 권 → "책장뷰 만들어드릴까요?"
원본 파일 업로드 직후 → "핵심 내용 카드뷰로 볼까요?"
노드 간 예상치 못한 연결 → "흥미로운 연결! 그래프로 볼까요?"
```

### 💡 AI 뷰 생성기 (Claude Code 연동)
```
OU가 LLM 서비스이기 때문에 가능한 것.
자연어 → Claude Code → HTML/CSS/JS 생성
→ DataNode 변수 자동 주입
→ 샌드박스 렌더링

"내 독서 기록을 책장 스타일로, 읽은 건 세워서, 안 읽은 건 눕혀서"
→ Claude Code가 생성 → 즉시 렌더링 → 수정 가능

이 뷰를 만드느라 토큰 소비
→ 자연스러운 Pro 플랜 과금 유도
만든 뷰를 마켓에 올려 판매 가능 → 크리에이터 이코노미
```

### 💡 토큰 단일 게이지 UX (과금 유도)
```
채팅 토큰 + 뷰 생성 토큰을 하나의 게이지로 표시
뷰 생성 시 게이지가 크게 줄어듦
→ "아 뷰 만드니까 빨리 줄어드네" 자연스럽게 인식
→ 업그레이드 동기 부여

뷰 생성 전: "이 작업은 채팅 약 N턴 분량을 사용해요" 안내
억지스럽지 않음. 가치를 느끼고 자연스럽게 결제.
```

### 💡 기본 DB 없어도 LLM이 채우되 신뢰도 차이 있음
```
기존 생각: 기본 DB가 있어야 정확한 DataNode 생성
수정된 생각:
  기본 DB 없어도 LLM이 어느 정도 채워줌 (confidence: 'medium')
  기본 DB 있으면 → confidence: 'high' (검증된 앵커)

기본 DB = LLM 추론의 정확도를 보증하는 앵커
         = 없어도 작동, 있으면 더 좋음
```

### 💡 검증 에이전트 (기본 DB 신뢰도 유지)
```
기본 DB도 시간이 지나면 outdated 됨
LLM이 채운 데이터는 hallucination 가능

주간 배치 검증 에이전트:
  외부 소스 대조 + LLM 재검증 + 트리플 일관성 검사
  의심 → 관리자 검토 큐
  관리자 승인 → confidence: 'high' 복원
```

### 💡 .ou 파일 (궁극적 목표)
```
Word가 .docx를 만들었듯 OU는 .ou를 만든다.

.docx = 문서 포맷
.ou   = 데이터 + 뷰의 분리된 포맷
        언어 중립, 무한 렌더링 가능

현재: 호환성 위해 PPT/Docs/PDF 익스포트
미래: .ou 하나로 모든 편집 완결
      굳이 다른 앱으로 가져갈 필요 없어짐
```

### 💡 OU = 인터넷 위의 공통 레이어
```
카톡/인스타/유튜브/쿠팡이 각각 만든 것들을
OU는 하나의 아키텍처 (DataNode + DataView)로 다 담는다.

카톡이 했던 것: 채팅 트래픽 기반 → 쇼핑/방송/결제 붙임
인스타가 했던 것: 이미지 DataNode → 광고 붙임

OU가 다 흡수할 수 있는 이유:
  트래픽 = 매일 쓰는 채팅창 + 매일 보는 피드
  거기에 DataNode 붙이면 어떤 사업이든 가능
  차이점: 모든 게 같은 아키텍처로 일관됨 (이질감 없음)
```

### 💡 OU의 락인이 기존 SNS보다 강력한 이유
```
기존 SNS 락인:
  내 데이터가 거기 갇혀 있어서 못 떠남 (억지)

OU 락인:
  내 데이터가 내 DB에 있음
  쌓을수록 그래프가 정교해지고 뷰가 많아지고
  연결이 늘어남
  이탈하면 이 우주를 잃는 것 (자발적)
  → 훨씬 강력한 락인
```

## 미래 확장 아이디어 (구현 순서 미정)

### 💡 쇼핑 DataNode + 구매뷰
```
상품 DataNode:
  {name, price, category, review, purchase_date}
구매뷰: 구매 내역 리스트
추천뷰: 관련 상품 그래프
카카오쇼핑 / 쿠팡 대체 가능
```

### 💡 방송/영상 DataNode + 방송뷰
```
영상 DataNode:
  {title, creator, timestamp, transcript, triples}
방송뷰: 유튜브/카카오TV 대체
수익화: 크리에이터가 영상 DataNode를 구독 판매
```

### 💡 결제 DataNode + 결제뷰
```
finance DataNode 이미 설계됨
결제뷰: 카카오페이/토스 대체
지출 패턴 그래프뷰: 소비 습관 분석
```

### 💡 라이브 커머스
```
영상 DataNode + 실시간 구매뷰
댓글 DataNode → 실시간 반응
구독자에게 라이브 알림
```

### 💡 익명 Q&A (교육 특화)
```
한국 교육 문화: 강의실에서 질문 거의 없음
익명으로 질문 → 다른 학생 upvote
AI가 정리 → 교수에게 주간 요약본 전송
→ 교수도 좋고, 학생도 좋음
→ 어느 에드테크도 제대로 못 푼 문제
(초기 대학 플랫폼 기획에서 등장한 아이디어)
```

### 💡 위치 DataNode + 지도뷰
```
{location, timestamp, context}
이동 기록 그래프뷰
약속 장소 DataNode → 지도뷰
카카오택시 대체 방향
```

### 💡 음악/웹툰 DataNode
```
음악 DataNode: {title, artist, genre, lyrics, mood}
  → 플레이어뷰 (카카오뮤직 대체)
  → 기분 기반 추천 뷰

웹툰 DataNode: {title, episode, author, genre}
  → 웹툰뷰 (카카오웹툰 대체)
```

### 💡 선물하기 = DataView 전송
```
내 DataNode로 만든 DataView를 선물로 전송
"내가 만든 독서 기록 뷰를 선물로 보내줄게"
→ 데이터가 선물이 됨
```

### 💡 브랜드 DataNode
```
기업이 제품/서비스 DataNode를 구축
→ 브랜드뷰 (브랜드 계정)
→ 소비자가 구독 → 신제품 알림
→ 인스타 브랜드 계정 대체
```

### 💡 학생 이해도 대시보드 (교수용)
```
초기 대학 플랫폼 아이디어에서 등장:
  수강생들이 어느 슬라이드에서 이해도가 떨어지는지
  교수가 대시보드로 확인
  → "교수님 자료를 올려주시면 학생 이해도 대시보드 드립니다"
  → 교수 온보딩 훅
```

### 💡 네트워크 효과 (같은 강의 수강생 데이터 축적)
```
같은 과목 수강생들의 DataNode가 쌓이면
→ 경쟁사가 따라올 수 없는 데이터 자산
"우리 학교 경영학원론 자료 1,200개 축적"
→ 신규 진입자가 만들 수 없음
```

### 💡 .ou 파일을 외부 앱들이 지원하게 되면
```
PDF가 표준이 된 것처럼
.ou가 새로운 문서 표준이 됨
외부 앱들이 .ou 열기/저장 지원
→ OU가 데이터 포맷 표준을 쥐게 됨
```

### 💡 IoT / 실시간 데이터 연동 (장기)
```
스마트워치 → 운동 DataNode 자동 생성
스마트홈 → 생활 패턴 DataNode
→ 입력 없이 자동으로 우주가 쌓임
```

### 💡 사용자가 직접 뷰 템플릿 제작 후 배포
```
AI 뷰 생성기로 만든 뷰를 마켓에 올림
다른 사용자가 구매 → 자신의 DataNode에 적용
크리에이터 이코노미의 핵심 상품
```

### 💡 그룹 DataNode 공개 → SNS 피드 자동 연결
```
그룹에서 만든 공동 DataNode를 공개 설정
→ SNS 피드에 자동 노출
→ 관심 있는 사람이 구독 or 그룹 참여 요청
→ 그룹 성장 플라이휠
```

## 아키텍처 원칙 (미래 확장을 위한)

```
지금 설계하는 코드가 미래 아이디어를 막으면 안 된다.

1. DataNode 스키마 확장 가능:
   domain 필드에 새 값 추가만으로 새 도메인 지원
   domain_data JSONB로 도메인별 구조화 데이터 유연하게

2. 뷰 레지스트리 플러그인 구조:
   새 뷰 타입 = 레지스트리에 등록만 하면 됨
   기존 코드 수정 불필요

3. 채팅/피드가 DataView 기반:
   어떤 DataNode든 뷰로 만들어 전송/표시 가능

4. 언어 중립 렌더링:
   DataNode 자체에 언어 없음
   렌더링 시 사용자 언어 설정 적용

5. 파일 원본 항상 보존:
   파싱/처리 로직이 바뀌어도 원본에서 재처리 가능
```

```
  → 전문가 집단의 자연스러운 참여
  → 관리자 부담 대폭 감소 (진짜 애매한 것만)
  → 커뮤니티 소속감 + 기여도
  → 미래: 검토 기여도 → 인센티브/배지/수익 연결

UX:
  OU 채팅/알림으로 자연스럽게 등장
  "잠깐, 이거 맞나요? 30초만요 🙏"
  객관식 선택 → 투표

DB: verification_requests + verification_votes
```

### 💡 데이터뷰 = 프론트엔드 (근본적 재정의)
```
컴퓨터 과학의 기본 개념:
  백엔드 (DB)     = 데이터
  프론트엔드 (UI) = 데이터를 보여주는 것
  = DataNode + DataView (완전히 동일한 구조)

OU가 새로 발명한 게 아니라 웹의 본질을 그대로 구현.

차이점 딱 하나:
  기존: 프론트엔드가 서비스에 종속
        인스타 프론트엔드 = 인스타 데이터만 보여줌
  OU:   DataView가 데이터에서 분리됨
        같은 DataNode → 어떤 DataView로든 렌더링 가능
        DataView 자체를 사고팔 수 있음
        DataView를 사용자가 직접 만들 수 있음

재정의:
  OU = 개인 DB + 그 DB의 프론트엔드를 무한으로 생성하는 플랫폼
```

### 💡 챗봇뷰 = 대화형 DataView (나를 학습한 AI)
```
기존 챗봇: 일반 데이터로 학습 → 일반적인 답변
챗봇뷰:    내 DataNode로 RAG → 나에 대한 답변
           = 나를 학습한 AI = 대화형 DataView

사용 사례:
  내 지식 DataNode → 챗봇뷰
    "나한테 뭐든 물어봐" → 내가 아는 것들로 답변

  강의 DataNode → 챗봇뷰
    수강생: "3강 내용 다시 설명해줘"
    → 강사 DataNode 기반 답변 → 강사가 24시간 답변하는 효과

  책 DataNode → 챗봇뷰
    "저자가 경제에 대해 뭐라고 했어?"
    → 책 DataNode 기반 답변

  기업 제품 DataNode → 챗봇뷰
    고객: "환불 정책이 뭐야?" → 제품 DataNode 기반 답변
    → 고객 서비스 챗봇 대체

  의사 진료 DataNode → 챗봇뷰
    환자: "내 지난 처방 기억해?" → 진료 기록 기반 답변

구독/판매:
  유명 한의사 → 한의학 DataNode + 챗봇뷰 유료 구독
  → 수강생이 "한의사에게 직접 질문"하는 경험
  저자 → 책 DataNode + 챗봇뷰
  → 독자가 "저자에게 직접 물어보기" 경험
  = 크리에이터 이코노미의 새로운 형태

기술:
  질문 → 관련 DataNode 벡터 검색 (RAG)
  → LLM이 내 DataNode 컨텍스트로 답변
  = DataView의 한 종류
```

```

               → 데이터를 보여주기만 함

               → 데이터를 시각적으로 표현

               → 데이터를 다른 포맷으로 변환

               → 데이터를 대화로 탐색 (RAG 기반)

               → AI 뷰 생성기(Claude Code)로 자동 생성
```

### 💡 이 대화 자체가 OU의 킬러 유즈케이스
```
페인포인트:
  기획 아이디어가 대화 중에 계속 나옴
  → Claude한테 "반영해줘" "기록해줘" 계속 요청해야 함
  → 반영됐는지 불안함
  → 대화가 휘발됨
  → 아이디어가 누락될까봐 답답함

OU였다면:
  이 대화 자체가 실시간으로 DataNode가 됨
  아이디어 나오는 족족 자동 분류
  → idea DataNode / knowledge DataNode
  → 그래프에 연결, 연결이 쌓임
  → "오늘 기획 대화에서 나온 아이디어 보여줘"
  → 전부 뷰로 렌더링, 문서로 출력

타겟 유저:
  기획자: 아이디어 대화 → 자동 기획서
  작가:   영감 대화 → 자동 메모 + 연결
  연구자: 논의 → 자동 논문 구조화
  창업자: 브레인스토밍 → 자동 사업계획

킬러 카피:
  "대화하는 족족 데이터가 됩니다"
  "아이디어는 사라지지 않아요"
  "지금 이 대화도 우주가 되고 있어요"

이것이 OU의 본질:
  LLM과의 대화 = 데이터 생산
  대화가 곧 기록이고, 기록이 곧 지식이고
  지식이 곧 우주가 된다
```

### 💡 가이드/설명서 에이전트 (서비스 이용 방법 자동 생성)
```
두 가지 역할:

A) 설명서 자동 생성
   서비스 기능 DataNode → 사용자 수준별 가이드 DataView 생성
   형태: 텍스트 / 스크린샷 / 인터랙티브 튜토리얼

   가이드 자체가 DataView:
     시스템 DataNode (기능 설명) → 가이드뷰로 렌더링

       beginner: 쉬운 말 + 스크린샷 위주
       expert:   간결한 텍스트 + 단축키 위주

B) UX 배치 결정
   사용자 행동 패턴 분석
   → 어떤 기능에서 막히는지 감지
   → 적절한 위치/시점에 가이드 노출

   배치 위치:
     툴팁:            버튼/아이콘 호버 시
     코치마크:        신규 기능 첫 진입 시
     모달:            복잡한 기능 처음 사용 시
     채팅 인라인:     "그거 어떻게 해요?"라고 물을 때
     홈 추천 카드:    아직 안 써본 기능 추천
     알림:            "이 기능 써보셨나요?"

트리거:
  신규 기능 배포 시 → 자동 가이드 생성
  사용자가 특정 기능에서 막힘 감지 → 컨텍스트 가이드 노출
  신규 가입 온보딩 → 단계별 가이드 생성

핵심 원칙:
  설명서를 만드는 게 아니라
  사용자가 막히는 순간 딱 맞는 것을 보여주는 것
  "어떻게 쓰는지 몰라서 안 쓴다"를 없애는 것
```
