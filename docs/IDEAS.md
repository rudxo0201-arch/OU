# IDEAS.md — 아이디어 로그 (누락 방지용)

> 이 파일은 기획 대화에서 나온 모든 인사이트와 아이디어를 맥락과 함께 기록한다.
> 당장 구현 안 해도 된다. 아이디어가 사라지면 안 된다.
> 구현 완료된 것은 ✅ 표시. 미구현은 💡 표시.

---

## 핵심 인사이트 로그

### 💡 LLM 대화가 데이터로 쌓이지 않는 문제
```
사람들은 LLM을 많이 쓴다. 대화하면서 인사이트를 얻는다.
그런데 이 대화가 좌측 패널에 시간순으로만 쌓인다.
→ 재탐색 불가, 연결 불가, 데이터로 활용 불가

OU의 출발점:
  대화 과정이 모두 데이터화되는 시스템
  세션 없음. 연속 스트림. 필터 + 뷰로 꺼내 씀.
```

### 💡 트리플화를 염두에 둔 LLM 생성 (온톨로지 해자)
```
일반 LLM: 생성 → 사후 트리플 추출 → 노이즈
OU LLM:  처음부터 표준 서술어 11개 안에서 생성
         → 온톨로지가 처음부터 정갈함
         → 데이터 쌓을수록 경쟁사가 따라올 수 없음
```

### 💡 ou-study: YouTube → DataNode 파이프라인
```
YouTube URL → 트랜스크립트 → sections → sentences → triples
현재 독립 Next.js 툴로 존재 (ou-study/)
→ OU 메인과 통합 예정
→ 관리자 기본 DB 구축의 주요 수단
```

### ✅ 데이터뷰 = 필터 × 템플릿
```
같은 데이터를 어떤 형태로든 꺼내 쓴다.
DataView = Filter × Template → Render
저장된 뷰 = 살아있는 뷰 (항상 최신 데이터 반영)
```

### 💡 프로필도 데이터뷰다 (SNS 정복 가능)
```
페이스북 프로필:
  이름/직장/학교/관계 DataNode → "프로필뷰"로 렌더링
  → 이게 전부다. DataNode + DataView.

인스타: 이미지 DataNode → 그리드뷰
트위터: 텍스트 DataNode → 피드뷰
링크드인: 경력 DataNode → 경력 프로필뷰

→ 기존 SNS가 하는 일 전부 = OU 아키텍처 안에 있음

OU 프로필:
  내 DataNode 중 공개 설정한 것들의 뷰
  지식 그래프뷰 / 관계 그래프뷰 / 타임라인뷰 / 포트폴리오뷰
  같은 나, 다른 뷰. 플랫폼마다 따로 관리 불필요.
```

### 💡 채팅 먹으면 → 카톡 사업 전체 흡수
```
카톡이 채팅 기반으로 붙인 것들:
  쇼핑 / 방송 / 결제 / 택시 / 웹툰 / 게임 / 선물하기

OU도 같은 이유로 가능:
  사람들이 채팅창에 매일 접속 = 트래픽의 중심
  거기에 붙이면 됨

차이점:
  카톡: 채팅앱에 쇼핑 억지로 붙임 (이질감)
  OU:   모든 게 DataNode + DataView로 일관됨
        쇼핑 = 상품 DataNode + 구매뷰 (이질감 없음)
```

### 💡 SNS 먹으면 → 광고시장 전체 흡수
```
SNS 광고시장이 거대한 이유:
  피드를 매일 봄 → 광고 노출
  인플루언서 팔로워 → 구독 기반 마케팅

OU SNS도 동일:
  피드뷰 → 인기 DataView에 광고
  구독 시스템 → 크리에이터 수익화
  라이브 커머스 → 영상 DataNode + 실시간 구매뷰

추가 장점:
  기존 SNS 광고: 플랫폼이 사용자 데이터를 팔아서 타겟팅
  OU 광고: 사용자 DataNode 기반 컨텍스트 광고
           → 더 정확한 타겟팅 (개인정보 직접 판매 없이)
```

### 💡 SNS 상호작용 자체가 DataNode
```
팔로우 = 구독 (이미 설계됨)
피드   = 구독한 DataNode 시간순 렌더링 (이미 설계됨)
좋아요 = {subject: 나, predicate: likes, object: 그 DataNode}
댓글   = 새 DataNode + 원본 DataNode와 relation 연결
공유   = DataView를 내 우주에 가져오기

→ 추가 설계할 게 없음. 피드뷰 + 프로필뷰만 추가하면 됨.
```

### 💡 데이터뷰가 강의/강의안이 될 수 있다 (교육 시장)
```
강의 DataNode → 강의뷰 → 수강생 구독
강사가 수정 → 즉시 반영
수강생은 자신의 노트 DataNode 추가
→ 강의 + 내 노트 = 하나의 우주

언어 중립:
  한국어 강의 → 외국 수강생에게 영어로 렌더링
  → 글로벌 강의 플랫폼 가능
```

### 💡 그래프뷰가 강의 콘텐츠 생성기다 (강사 플라이휠)
```
강사가 DataNode를 쌓으면
그래프뷰가 강사도 몰랐던 연결을 보여준다
→ 새 강의 주제 무한 생성
→ 더 많이 쌓고 싶어짐 → 플라이휠

예: 한의사가 황기/인삼 DataNode 쌓음
    → "보기" 클러스터 → "보기약 비교 강의" 주제 발견
    → "폐" 중심 엣지 → "폐 관련 본초 총정리" 발견
```

### 💡 문제지 시장도 다 흡수 가능
```
현재 수학 문제지 시장 유행:
  단원/난이도 선택 → PDF 자동 생성 → 끝

OU:
  문제 DataNode → 어떤 뷰로든:
    PDF뷰: 문제지
    퀴즈뷰: 인터랙티브 풀기
    오답 DataNode: 자동 생성
    오답 PDF뷰: 오답노트
    취약점 그래프뷰: "내가 틀리는 유형"
    맞춤 문제지: 취약 단원만 추출 → PDF

데이터는 하나, 뷰는 무한.
풀수록 데이터 쌓임 → 더 정교한 맞춤 문제지.
```

### 💡 원본 파일 보존 필수 (양방향 연결)
```
PDF A → DataNode 생성
→ 원본 PDF A와 연결되어야 함
→ DataNode에서 "원본 보기" 가능
→ 해당 페이지/섹션으로 자동 스크롤

파싱 실패해도 원본은 R2에 보존 (재시도 가능)
원본 손실은 복구 불가 → 원본 보존이 절대 최우선

모든 파일 형식이 OU 안에서 렌더링될 수 있어야 한다.
→ .ou 시대의 전제 조건
```

### 💡 데이터가 쌓이면 뷰를 추천해줘야 한다
```
사용자는 뷰를 몰라도 된다.
데이터가 쌓이면 OU가 먼저 제안한다.

일정 5개 → "캘린더뷰 어떨까요?"
독서 여러 권 → "책장뷰 만들어드릴까요?"
원본 파일 업로드 직후 → "핵심 내용 카드뷰로 볼까요?"
노드 간 예상치 못한 연결 → "흥미로운 연결! 그래프로 볼까요?"
```

### 💡 AI 뷰 생성기 (Claude Code 연동)
```
OU가 LLM 서비스이기 때문에 가능한 것.
자연어 → Claude Code → HTML/CSS/JS 생성
→ DataNode 변수 자동 주입
→ 샌드박스 렌더링

"내 독서 기록을 책장 스타일로, 읽은 건 세워서, 안 읽은 건 눕혀서"
→ Claude Code가 생성 → 즉시 렌더링 → 수정 가능

이 뷰를 만드느라 토큰 소비
→ 자연스러운 Pro 플랜 과금 유도
만든 뷰를 마켓에 올려 판매 가능 → 크리에이터 이코노미
```

### 💡 AI = 뷰의 발명자 (세상에 없던 뷰 생성)
```
핵심 인사이트:
  인류가 만든 모든 뷰(표, 차트, 칸반, 피드)는
  "그 시대 기술 한계 안에서 나온 것"일 뿐.
  데이터를 가장 잘 보여주는 방식이라는 보장이 없다.

  표    → 종이 시대의 산물
  차트  → 통계학의 산물
  칸반  → 도요타 공장의 산물
  피드  → 트위터의 산물

AI 뷰 생성기의 진짜 가치:
  "기존 뷰 중 골라주는 것"이 아니라
  "이 데이터에 최적인 뷰를 발명하는 것"

  AI가 데이터의 구조, 관계, 맥락을 분석 →
  기존에 없던 시각화 형태를 HTML/CSS/JS로 생성 →
  형태에 제한 없음 (웹 기술이 곧 캔버스)

예시 (존재하지 않는 뷰):
  감정 데이터 → 차트 대신 "파동 뷰" (리듬과 진폭으로 표현)
  학습 진도 → 퍼센트 바 대신 "지형도 뷰" (높낮이 = 이해도)
  관계 데이터 → 리스트 대신 "중력 뷰" (가까울수록 친밀)
  재무 데이터 → 막대 차트 대신 "생태계 뷰" (수입=비, 지출=증발)
  습관 데이터 → 히트맵 대신 "성장 뷰" (나무가 자라는 형태)

OU = 뷰의 소비자가 아니라 뷰의 생산자.
     세상에 없던 시각화가 OU에서 탄생한다.
     차트보다 직관적인 무언가가 나올 수 있다.

이것이 가능한 이유:
  1. DataNode가 구조화되어 있음 (도메인, 트리플, 벡터)
     → AI가 데이터 특성을 정확히 파악 가능
  2. HTML/CSS/JS로 렌더링 = 표현의 제약 없음
     → 2D, 3D, 애니메이션, 인터랙션 모두 가능
  3. AI가 수만 가지 시각화 논문/사례를 학습함
     → 인간 디자이너보다 넓은 레퍼런스 풀
  4. 사용자 피드백 루프
     → "이거 좀 다르게" → 즉시 변형 → 진화

미래:
  OU에서 탄생한 뷰가 "표준 뷰"로 채택되는 날이 온다.
  차트가 통계학에서 나왔듯, 새로운 시각화 패러다임이 OU에서 나온다.
```

### 💡 토큰 단일 게이지 UX (과금 유도)
```
채팅 토큰 + 뷰 생성 토큰을 하나의 게이지로 표시
뷰 생성 시 게이지가 크게 줄어듦
→ "아 뷰 만드니까 빨리 줄어드네" 자연스럽게 인식
→ 업그레이드 동기 부여

뷰 생성 전: "이 작업은 채팅 약 N턴 분량을 사용해요" 안내
억지스럽지 않음. 가치를 느끼고 자연스럽게 결제.
```

### 💡 기본 DB 없어도 LLM이 채우되 신뢰도 차이 있음
```
기존 생각: 기본 DB가 있어야 정확한 DataNode 생성
수정된 생각:
  기본 DB 없어도 LLM이 어느 정도 채워줌 (confidence: 'medium')
  기본 DB 있으면 → confidence: 'high' (검증된 앵커)

기본 DB = LLM 추론의 정확도를 보증하는 앵커
         = 없어도 작동, 있으면 더 좋음
```

### 💡 검증 에이전트 (기본 DB 신뢰도 유지)
```
기본 DB도 시간이 지나면 outdated 됨
LLM이 채운 데이터는 hallucination 가능

주간 배치 검증 에이전트:
  외부 소스 대조 + LLM 재검증 + 트리플 일관성 검사
  의심 → 관리자 검토 큐
  관리자 승인 → confidence: 'high' 복원
```

### 💡 .ou 파일 (궁극적 목표)
```
Word가 .docx를 만들었듯 OU는 .ou를 만든다.

.docx = 문서 포맷
.ou   = 데이터 + 뷰의 분리된 포맷
        언어 중립, 무한 렌더링 가능

현재: 호환성 위해 PPT/Docs/PDF 익스포트
미래: .ou 하나로 모든 편집 완결
      굳이 다른 앱으로 가져갈 필요 없어짐
```

### 💡 OU = 인터넷 위의 공통 레이어
```
카톡/인스타/유튜브/쿠팡이 각각 만든 것들을
OU는 하나의 아키텍처 (DataNode + DataView)로 다 담는다.

카톡이 했던 것: 채팅 트래픽 기반 → 쇼핑/방송/결제 붙임
인스타가 했던 것: 이미지 DataNode → 광고 붙임

OU가 다 흡수할 수 있는 이유:
  트래픽 = 매일 쓰는 채팅창 + 매일 보는 피드
  거기에 DataNode 붙이면 어떤 사업이든 가능
  차이점: 모든 게 같은 아키텍처로 일관됨 (이질감 없음)
```

### 💡 OU의 락인이 기존 SNS보다 강력한 이유
```
기존 SNS 락인:
  내 데이터가 거기 갇혀 있어서 못 떠남 (억지)

OU 락인:
  내 데이터가 내 DB에 있음
  쌓을수록 그래프가 정교해지고 뷰가 많아지고
  연결이 늘어남
  이탈하면 이 우주를 잃는 것 (자발적)
  → 훨씬 강력한 락인
```

## 🌍 패러다임 전환 — OU가 대체하는 것들

> OU는 누구와도 경쟁하지 않지만, 모든 서비스를 대체한다.
> 개별 SaaS를 이기는 게 아니라, "데이터를 형태에 가두는 방식" 자체가 사라진다.
> 아래는 OU가 대체하는 기존 패러다임과 그 이유.

### 💡 OS / 파일시스템 대체
```
현재: 폴더 → 파일 → 앱으로 열기 (1980년대 패러다임)
      "어디에 저장했지?" = 폴더 구조에 갇힌 사고방식
      .docx는 Word로, .ppt는 파워포인트로, .pdf는 뷰어로
      → 파일 형식이 앱을 결정함

OU:   데이터가 들어오면 형태가 없음. 뷰가 형태를 만들어줌
      파일 확장자, 폴더 구조, "어디에 저장했지?" 자체가 사라짐
      같은 데이터 → 문서로도, 표로도, 그래프로도, 발표자료로도

구현 기반: source_file_url (R2), source_file_type, DataNode 자동 변환
→ 파일 업로드 시 DataNode화 파이프라인 이미 존재
→ 파일 탐색기 뷰 = DataNode 필터(source_file_type) + 리스트/그리드 뷰
```

### 💡 검색엔진 대체
```
현재: Google = 남의 데이터에서 키워드로 찾기
      검색 결과 = 남이 만든 페이지 링크 목록
      내 맥락과 무관한 결과

OU:   내 데이터 안에서 관계로 찾기
      "이거랑 연결된 거 뭐 있어?" = 그래프 탐색
      "비슷한 거 뭐 있어?" = 벡터 유사도 검색
      검색이 아니라 탐색. 발견.

구현 기반: sentences.embedding (벡터), triples (관계), node_relations (엣지)
→ 시맨틱 검색 (벡터) + 그래프 탐색 (관계) 조합
→ 검색 결과 자체가 DataView (리스트/그래프/맵 등)
```

### 💡 개인의 기억 외부화
```
현재: 기억은 뇌에 의존. 흩어진 앱에 파편화.
      "작년에 그거 뭐였지?" → 카톡/메모/사진첩/캘린더 뒤짐
      찾아도 맥락이 없음. 연결이 없음.

OU:   대화, 메모, 사진, 강의, 문서... 모든 정보가 DataNode
      시간축으로 보면 타임라인. 주제별로 보면 그래프.
      "작년 봄에 뭐했지?" → Just talk → 즉시 뷰

구현 기반: messages → data_nodes 파이프라인, created_at 타임스탬프
→ 타임라인뷰 (이미 구현), 시간여행 뷰 (시점별 그래프 스냅샷)
→ 모든 입력이 자동 DataNode화 = 기억의 자동 외부화
```

### 💡 조직의 암묵지 구조화
```
현재: 퇴사자 머릿속 노하우 = 회사 최대 손실
      Slack/Notion에 있어도 구조화 안 됨 → 못 찾음
      신입이 1년 걸려 파악하는 것 = 암묵지

OU:   업무 대화가 자동으로 DataNode화
      트리플로 관계 구조화 → 그래프로 시각화
      사람이 떠나도 지식 그래프는 남음
      신입: "이 프로젝트 맥락 알려줘" → 챗봇뷰로 즉답

구현 기반: group_id (그룹 소속), visibility (공유 범위), 챗봇뷰 (RAG)
→ 그룹 지식 그래프뷰 = group_id 필터 + knowledge_graph 뷰
→ 그룹 챗봇뷰 = 그룹 DataNode 기반 RAG
```

### 💡 교육 시스템 재정의
```
현재: 교과서 = 고정된 뷰. 모든 학생에게 같은 순서, 같은 깊이.
      시험 = 고정된 문제. 이해 여부와 무관한 평가.
      강의 = 1:N 단방향. 질문 어려움.

OU:   같은 지식 그래프를 학생마다 다른 뷰로 봄
      이미 아는 건 축소, 약한 부분은 확장
      교과서가 아니라 "적응형 지식 우주"

구현 기반: education 도메인, lecture/flashcard/quiz 뷰 이미 존재
→ 적응형 로직 = 오답 DataNode 분석 → 취약 영역 자동 확장
→ 학생 이해도 대시보드 = 교수용 뷰
```

### 💡 미디어 / 저널리즘 재정의
```
현재: 기사 = 기자가 만든 고정 뷰. 하나의 관점.
      뉴스 = 시간순 피드. 맥락 없음.
      "이 사건의 전체 그림"을 보려면 직접 여러 기사 조합

OU:   팩트가 DataNode. 기사는 하나의 뷰일 뿐.
      같은 사건 → 경제 관점 뷰, 기술 관점 뷰, 사회 관점 뷰
      시간축 → 타임라인뷰. 관련 사건 → 그래프뷰.
      독자가 자기 관점의 뷰를 직접 만들 수 있음

구현 기반: knowledge/media 도메인, 다중 뷰 렌더링
→ 뉴스 DataNode + 관점별 필터 = 관점별 뷰
→ 팩트 그래프 = 사건 간 트리플 연결
```

### 💡 스프레드시트 / 데이터 도구 대체
```
현재: Excel = 세상에서 가장 많이 쓰이는 데이터 도구
      BUT 행×열에 갇혀 있음. 시각화하려면 차트 따로.
      다른 사람과 공유하려면 파일 전송.
      피벗 테이블 = 같은 데이터를 다른 뷰로 보는 원시적 형태

OU:   테이블뷰가 DataView 중 하나일 뿐
      같은 데이터 → 테이블, 차트, 리포트, 대시보드
      공유 = 뷰 링크 하나. 실시간 반영.
      수식 = 트리플 관계 + LLM 연산

구현 기반: table 뷰 + chart 뷰 이미 레지스트리에 존재
→ 데이터 편집 가능한 테이블뷰 = domain_data JSONB 편집
→ 피벗/집계 = filter_config로 그룹핑
```

### 💡 클라우드 스토리지 / 협업 도구 대체
```
현재: Dropbox/Drive = 파일을 폴더에 넣고 꺼내는 구조
      Notion = 문서 기반 협업. 도구마다 데이터 사일로.
      Slack = 대화와 파일이 흘러가고 사라짐

OU:   파일이든 텍스트든 대화든 전부 DataNode
      폴더 계층 대신 그래프 관계로 정리
      같은 데이터를 누구는 칸반, 누구는 타임라인, 누구는 채팅으로
      대화 자체가 데이터화 → 사라지지 않음

구현 기반: group_id + visibility + saved_views + view_members
→ 협업 = 같은 DataNode에 다른 뷰. 권한별 필터.
→ 클라우드 = source_file_url (R2) + DataNode 메타데이터
```

### 💡 SNS 자동화 도구 불필요
```
현재: 인스타/스레드/유튜브용 콘텐츠를 따로 만듦
      Hootsuite, Buffer 같은 자동화 툴로 각 채널에 배포
      채널마다 포맷 다름 → 같은 내용을 N번 가공

OU:   데이터를 한번 넣으면 SNS 뷰가 각 채널 포맷으로 뽑아줌
      인스타 = 이미지 그리드 뷰
      스레드 = 텍스트 피드 뷰
      유튜브 = 영상 + 자막 뷰
      자동화 툴이 아니라 DataView 자체가 배포 채널

구현 기반: SNS 피드뷰 + visibility:'public' + 외부 API 연동
→ 현재: 내부 SNS 피드뷰 구현 가능
→ 미래: 외부 채널 API 연동으로 원클릭 배포
```

## 아키텍처 원칙 (미래 확장을 위한)

```
지금 설계하는 코드가 미래 아이디어를 막으면 안 된다.

1. DataNode 스키마 확장 가능:
   domain 필드에 새 값 추가만으로 새 도메인 지원
   domain_data JSONB로 도메인별 구조화 데이터 유연하게

2. 뷰 레지스트리 플러그인 구조:
   새 뷰 타입 = 레지스트리에 등록만 하면 됨
   기존 코드 수정 불필요

3. 채팅/피드가 DataView 기반:
   어떤 DataNode든 뷰로 만들어 전송/표시 가능

4. 언어 중립 렌더링:
   DataNode 자체에 언어 없음
   렌더링 시 사용자 언어 설정 적용

5. 파일 원본 항상 보존:
   파싱/처리 로직이 바뀌어도 원본에서 재처리 가능
```

### 💡 집단지성 검토 시스템 (관리자 병목 해소)
```
기존: 관리자 혼자 의심 DataNode 검토 → 스케일 불가

개선: 집단지성 크라우드소싱
  대상자: 해당 DataNode 관련 DataNode를 가진 회원
          = 이 데이터에 실제로 관심 있는 사람
  방식: 객관식 투표 (맞아요 / 틀려요 / 모르겠어요)
  임계값: N명 "맞아요" → confidence = 'high' 자동 복원
          N명 "틀려요" → 관리자 최종 검토 큐

핵심 장점:
  한의학 DataNode → 한의사/한의대생이 검토 (관리자보다 정확)
  의학 DataNode → 의사/의대생이 검토
  → 전문가 집단의 자연스러운 참여
  → 관리자 부담 대폭 감소 (진짜 애매한 것만)
  → 커뮤니티 소속감 + 기여도
  → 미래: 검토 기여도 → 인센티브/배지/수익 연결

UX:
  OU 채팅/알림으로 자연스럽게 등장
  "잠깐, 이거 맞나요? 30초만요 🙏"
  객관식 선택 → 투표

DB: verification_requests + verification_votes
```

### 💡 데이터뷰 = 프론트엔드 (근본적 재정의)
```
컴퓨터 과학의 기본 개념:
  백엔드 (DB)     = 데이터
  프론트엔드 (UI) = 데이터를 보여주는 것
  = DataNode + DataView (완전히 동일한 구조)

OU가 새로 발명한 게 아니라 웹의 본질을 그대로 구현.

차이점 딱 하나:
  기존: 프론트엔드가 서비스에 종속
        인스타 프론트엔드 = 인스타 데이터만 보여줌
  OU:   DataView가 데이터에서 분리됨
        같은 DataNode → 어떤 DataView로든 렌더링 가능
        DataView 자체를 사고팔 수 있음
        DataView를 사용자가 직접 만들 수 있음

재정의:
  OU = 개인 DB + 그 DB의 프론트엔드를 무한으로 생성하는 플랫폼
```

### 💡 챗봇뷰 = 대화형 DataView (나를 학습한 AI)
```
기존 챗봇: 일반 데이터로 학습 → 일반적인 답변
챗봇뷰:    내 DataNode로 RAG → 나에 대한 답변
           = 나를 학습한 AI = 대화형 DataView

사용 사례:
  내 지식 DataNode → 챗봇뷰
    "나한테 뭐든 물어봐" → 내가 아는 것들로 답변

  강의 DataNode → 챗봇뷰
    수강생: "3강 내용 다시 설명해줘"
    → 강사 DataNode 기반 답변 → 강사가 24시간 답변하는 효과

  책 DataNode → 챗봇뷰
    "저자가 경제에 대해 뭐라고 했어?"
    → 책 DataNode 기반 답변

  기업 제품 DataNode → 챗봇뷰
    고객: "환불 정책이 뭐야?" → 제품 DataNode 기반 답변
    → 고객 서비스 챗봇 대체

  의사 진료 DataNode → 챗봇뷰
    환자: "내 지난 처방 기억해?" → 진료 기록 기반 답변

구독/판매:
  유명 한의사 → 한의학 DataNode + 챗봇뷰 유료 구독
  → 수강생이 "한의사에게 직접 질문"하는 경험
  저자 → 책 DataNode + 챗봇뷰
  → 독자가 "저자에게 직접 물어보기" 경험
  = 크리에이터 이코노미의 새로운 형태

기술:
  질문 → 관련 DataNode 벡터 검색 (RAG)
  → LLM이 내 DataNode 컨텍스트로 답변
  = DataView의 한 종류
```

### 💡 전자책뷰 / 출판책뷰
```
전자책뷰:
  knowledge DataNode들
  → 챕터 구조 레이아웃
  → 페이지 넘김 인터랙션
  → 폰트/여백/색상 커스텀
  → .epub 익스포트

출판책뷰:
  동일한 DataNode
  → 인쇄 레이아웃 (A5, B5 등)
  → 페이지 번호 / 목차 / 헤더·푸터 자동 생성
  → PDF 익스포트 → 실제 인쇄 가능
  → POD (Print on Demand) 연결 가능

같은 DataNode 하나로:
  전자책 (.epub) / 출판책 (PDF) / 웹 강의 / 문제집 / 챗봇
  = 같은 데이터, 다른 프론트엔드
```

### 💡 DataView 전체 재분류 (프론트엔드 관점)
```
정적 뷰:       표, 리스트, 칸반, 캘린더
               → 데이터를 보여주기만 함

시각화 뷰:     그래프뷰, 차트, 마인드맵
               → 데이터를 시각적으로 표현

출력 뷰:       PDF, PPT, 전자책(.epub), 출판책
               → 데이터를 다른 포맷으로 변환

인터랙티브 뷰: 퀴즈뷰, 플래시카드뷰
               → 데이터와 상호작용

대화형 뷰:     챗봇뷰
               → 데이터를 대화로 탐색 (RAG 기반)

웹 뷰:         HTML/CSS/JS 프론트엔드
               → 데이터를 웹사이트로 표현
               → AI 뷰 생성기(Claude Code)로 자동 생성

모든 게 프론트엔드고, 모든 게 DataView.
```

```

### 💡 이 대화 자체가 OU의 킬러 유즈케이스
```
페인포인트:
  기획 아이디어가 대화 중에 계속 나옴
  → Claude한테 "반영해줘" "기록해줘" 계속 요청해야 함
  → 반영됐는지 불안함
  → 대화가 휘발됨
  → 아이디어가 누락될까봐 답답함

OU였다면:
  이 대화 자체가 실시간으로 DataNode가 됨
  아이디어 나오는 족족 자동 분류
  → idea DataNode / knowledge DataNode
  → 그래프에 연결, 연결이 쌓임
  → "오늘 기획 대화에서 나온 아이디어 보여줘"
  → 전부 뷰로 렌더링, 문서로 출력

타겟 유저:
  기획자: 아이디어 대화 → 자동 기획서
  작가:   영감 대화 → 자동 메모 + 연결
  연구자: 논의 → 자동 논문 구조화
  창업자: 브레인스토밍 → 자동 사업계획

킬러 카피:
  "대화하는 족족 데이터가 됩니다"
  "아이디어는 사라지지 않아요"
  "지금 이 대화도 우주가 되고 있어요"

이것이 OU의 본질:
  LLM과의 대화 = 데이터 생산
  대화가 곧 기록이고, 기록이 곧 지식이고
  지식이 곧 우주가 된다
```

B) UX 배치 결정

트리거:
  신규 기능 배포 시 → 자동 가이드 생성
  사용자가 특정 기능에서 막힘 감지 → 컨텍스트 가이드 노출
  신규 가입 온보딩 → 단계별 가이드 생성

핵심 원칙:
  설명서를 만드는 게 아니라
  사용자가 막히는 순간 딱 맞는 것을 보여주는 것

## PDF 데이터화 엔진 설계 (구현 확정)

> Phase 1 + Phase 2로 나누어 구현. 정확도와 품질 최우선.

### ✅ 핵심 원칙 (불변)
```
- order_idx는 문서 전체 기준 (페이지 기준 아님)
- 하나의 문장이 두 개의 order_idx로 쪼개지면 안 됨
- 이미지도 데이터 — 장식은 버리고 의미 있는 이미지만 DB화
- 깨진 PDF는 버리지 않고 보정한다
```

### 💡 Phase 1: 텍스트 파싱 엔진

#### 파싱 파이프라인 (pdfjs-dist 기반)
```
Step 1: 페이지별 텍스트 + 메타데이터 추출
  pdfjs-dist getPage() → getTextContent()
  각 아이템: text, fontSize, fontName, x, y, width, height

Step 1.5: 쪽번호 + 러닝헤더 분리
  y좌표 최상단/최하단 영역 텍스트 분리
  - 숫자 단독/"- N -"/"p.N" → 쪽번호
  - 모든 페이지 같은 y좌표 반복 → 러닝헤더
  → 본문에서 제거, 메타데이터로 저장: { pageIndex, pageLabel }

Step 2: 헤더 감지
  1차: getOutline() → bookmark 트리 (계층 구조 보존)
  2차 폴백: 폰트 크기 + 볼드 + "제N장"/"Chapter"/"N." 패턴

Step 3: 페이지 간 텍스트 병합 (문장 분리 방지 — 핵심)
  전체 문서를 하나의 텍스트 스트림으로 병합한 뒤 sentence 분할.
  페이지 경계는 sentence 분할에 개입하지 않음.

  Case A: 다음 페이지가 헤더로 시작
    → 이전 텍스트 종결 (불완전해도 독립 sentence)
    → 새 section 시작

  Case B: 이전 페이지 끝이 종결부호(.!?。)
    → 자연 분리

  Case C: 문장 중간 끊김 (핵심)
    → 다음 페이지 텍스트와 이어붙여 하나의 sentence로 유지
    → pages: [N, N+1] 부여
    → 절대 두 개의 order_idx로 쪼개지지 않음

  구현: 페이지별 분할 금지. 전체 병합 → 분할 → 역매핑.

Step 4: section/sentence 분할 (병합된 전체 텍스트 대상)
Step 5: 페이지 역매핑 (source_location 메타데이터)
```

#### 추가 처리
```
다단 레이아웃: x좌표 클러스터링 → 단별 순차 읽기
표(Table): 좌표 그리드 감지 → Gemini Vision 구조화
각주: 하단 영역 + 작은 폰트 + 숫자 패턴 → 분리 + 참조 연결
스캔 PDF: 텍스트 레이어 비어있으면 → 전체 Gemini Vision OCR
```

#### 깨진 PDF 보정 (3단계, 자기강화 루프)
```
1단계: 품질 자동 진단 (무료)
  한글/영문 vs 깨진 유니코드 비율, 단어 길이, 텍스트량/페이지수
  → confidence: high / medium / low

2단계: low → Gemini Vision OCR 재추출
  텍스트 레이어 폐기 → 페이지 이미지로 렌더링 → OCR

3단계: 도메인 메타데이터 LLM 보정
  파일명에서 도메인 힌트 + DB 내 관련 트리플 검색
  → LLM에 컨텍스트 제공 → 깨진 텍스트 문맥 복원
  → OU 데이터가 쌓일수록 보정 정확도 ↑ (자기강화 루프)
```

### 💡 Phase 1: 이미지 DB화

#### 추출 + 필터링
```
1단계: 임베디드 이미지 → 원본 바이너리 추출 (pdfjs-dist operatorList)
2단계: 규칙 필터 (장식 제거)

저장: R2에 {userId}/nodes/{nodeId}/img-{order}.png
