# OU (OWN UNIVERSE) — 사업 소개

---

## 한 줄 정의

**LLM과의 대화를 자동으로 구조화하고, 어떤 형태로든 꺼내 쓸 수 있는 개인 데이터 플랫폼.**

---

## 1. 문제 인식 — 데이터는 넘치는데, 구조가 없다

사람들은 이미 ChatGPT, Claude 같은 LLM을 매일 쓴다. 일정을 물어보고, 아이디어를 정리하고, 업무를 논의한다.

그런데 이 대화는 **좌측 패널에 시간순으로 쌓일 뿐이다.**

- "3주 전에 AI랑 나눈 사업 아이디어가 뭐였지?" → 못 찾는다
- "지난달에 정리한 독서 노트 어디 갔지?" → 스크롤해서 뒤져야 한다
- 노션이나 옵시디언에 옮겨도 **텍스트 더미**일 뿐, 관계가 없다

핵심 문제: **비정형 데이터가 구조화되지 않으면, 양이 아무리 많아도 자산이 아니다.**

---

## 2. OU의 해법 — 대화가 곧 데이터베이스

OU에서는 사용자가 채팅창에 입력하기만 하면 된다.

```
"다음주 일요일 희민이 결혼식" → 캘린더에 일정 노드 생성
"커피 5000원"                → 가계부에 지출 노드 생성
"재미있는 앱 아이디어 떠올랐어" → 아이디어 마인드맵에 노드 생성
```

**입력은 자연어, 저장은 그래프 데이터베이스.**

내부적으로 일어나는 일:

1. **도메인 분류**: 입력 텍스트를 정규식 + 패턴 매칭으로 분류 (일정/할일/재정/감정/지식/아이디어 등 14개 도메인). LLM 호출 없이, 비용 0.

2. **구조화**: 각 입력이 DataNode(노드)가 되고, 내부적으로 section → sentence 계층으로 분해된다.

3. **트리플 추출**: (subject, predicate, object) 형태의 온톨로지 트리플을 자동 생성한다. 서술어는 11개로 고정 (`is_a`, `part_of`, `causes`, `derived_from`, `related_to`, `opposite_of`, `requires`, `example_of`, `involves`, `located_at`, `occurs_at`). 이 제약이 스키마의 일관성을 보장한다.

4. **임베딩**: 문장 단위로 벡터 임베딩(OpenAI text-embedding-3-small, 1536차원)하여 의미 검색을 가능하게 한다.

5. **그래프 연결**: 트리플이 노드 간 엣지가 되어, 데이터가 쌓일수록 관계망이 자기강화된다.

---

## 3. DataView — 같은 데이터, 무한한 표현

OU의 핵심 차별점은 **데이터와 표현의 완전한 분리**다.

하나의 DataNode를 어떤 뷰로든 렌더링할 수 있다:

| 같은 데이터 | → 뷰 A | → 뷰 B | → 뷰 C |
|---|---|---|---|
| 일정 노드 | 캘린더뷰 | 타임라인뷰 | 리스트뷰 |
| 지식 노드 | 그래프뷰 | 플래시카드뷰 | 문서뷰 |
| 코드 파일 | 파일뷰 (에디터) | 그래프뷰 (의존성) | 타임라인뷰 (변경 이력) |

이것은 컴퓨터 과학의 근본 구조와 동일하다:
- **백엔드(DB)** = 데이터 = DataNode
- **프론트엔드(UI)** = 표현 = DataView

OU는 이 구조를 **개인이 소유하고, 커스텀하고, 거래할 수 있게** 만든다.

뷰는 플러그인 구조(레지스트리 패턴)로, 새 뷰를 등록하면 기존 코드 수정 없이 확장된다. 현재 20종 이상의 뷰가 구현되어 있다.

---

## 4. 아키텍처 — 왜 이 구조가 강력한가

```
사용자 입력 (자연어)
    ↓
Layer 1: 파싱 (PDF/이미지/텍스트/코드 → 구조화)
    ↓
Layer 2: 분류 + 도메인 데이터 추출 (비용 0, 정규식 기반)
    ↓
Layer 3: 비동기 강화 (임베딩 + 트리플 추출 + 이미지 분석)
    ↓
Graph DB (PostgreSQL + pgvector)
    ↓
DataView (무한 뷰 렌더링)
```

**설계 원칙:**
- **세션 없음**: 대화는 연속 스트림. "대화 1", "대화 2" 같은 인위적 경계가 없다.
- **데이터 불삭제**: 모든 변경은 이벤트로 기록. 삭제는 아카이브.
- **원본 보존**: 파싱이 실패해도 원본은 반드시 저장 (Cloudflare R2).
- **비용 최적화**: Layer 2까지 LLM 호출 0. Layer 3는 비동기이며, 데이터 온도(hot/warm/cold)에 따라 임베딩 비용을 동적 조절.

---

## 5. 철학 — 파일은 사라진다

OU의 근본적 주장:

> **파일과 폴더는 컴퓨터의 한계를 인간에게 떠넘긴 것이다.**

사람의 뇌에는 파일이 없다. "약속 → 그 사람 → 지난번 대화 → 거기서 나온 아이디어" — 이것이 하나의 네트워크로 연결되어 있다. 앱 경계도, 폴더 구분도 없다.

OU는 데이터를 **원래 형태 — 인간의 사고 구조 — 로 되돌린다.**

- 인스타그램이 하는 일: 이미지 데이터 → 그리드뷰로 렌더링
- 트위터가 하는 일: 텍스트 데이터 → 피드뷰로 렌더링
- 링크드인이 하는 일: 경력 데이터 → 프로필뷰로 렌더링
- **모두 DataNode + DataView다. OU는 이것의 범용 플랫폼이다.**

---

## 6. 해자(Moat) — 쓸수록 이탈이 불가능해진다

기존 서비스의 락인: 플랫폼이 데이터를 인질로 잡는다.

OU의 락인: **사용자가 자발적으로 자신의 우주를 키운다.** 데이터는 사용자 소유이지만, 이 그래프 — 수년간 쌓인 지식, 관계, 맥락의 연결망 — 을 다른 플랫폼으로 가져갈 수 없다. 노드 하나하나는 텍스트지만, **노드 간 관계(트리플)가 진짜 자산**이기 때문이다.

비유하자면:
- 노트앱에서 이탈 = 텍스트 파일 복사하면 끝
- OU에서 이탈 = **뇌를 복사해야 하는 것**

---

## 7. 사업 구조 — DataNode + DataView가 흡수하는 시장

OU의 아키텍처는 특정 시장을 겨냥한 것이 아니다. **DataNode + DataView 구조가 자연스럽게 확장되는 것**이다.

```
1단계: LLM 채팅 → 일정앱/노트앱/가계부/습관앱 대체
2단계: 그룹/구독 시스템 → 메신저/단톡방 대체
3단계: SNS 피드/프로필 → 소셜 미디어 대체 + 광고 시장 흡수
4단계: 교육 시장 → 강의/문제집/오답노트 (뷰로 표현)
5단계: 개발 환경 → 코드도 DataNode, 개발 과정이 데이터가 됨
6단계: 커머스/콘텐츠/금융 → 모든 것이 DataNode
```

핵심: 단계마다 기능을 억지로 붙이는 것이 아니라, **같은 아키텍처가 도메인만 바꿔서 자연스럽게 확장**된다.

## 8. 수익 모델

1. **구독**: 무료(제한된 노드) → 프로 → 팀
2. **DataView 마켓**: 사용자가 만든 뷰를 거래 (OU가 수수료)
3. **교육 시장**: 대학/학원이 OU 위에서 강의 운영 (B2B)
4. **데이터 라이선싱**: 출판사 원본 데이터를 계약으로 확보, 고품질 DataNode로 제공
5. **광고**: SNS 단계에서 컨텍스트 기반 광고 (개인정보 직접 판매 없이)

## 10. OU가 궁극적으로 만드는 것

**1명의 OU = 개인 뇌의 디지털 복제.**
**N명의 OU = N개의 뇌가 연결된 네트워크.**

현재 AI는 웹페이지와 문서 — 납작한 텍스트 — 로 학습한다. 관계를 텍스트에서 추론해야 한다. 도서관의 책을 전부 읽은 것이지, 경험이 없다.

OU의 데이터로 학습하면:
- 사람의 실제 사고 흐름이 **구조화되어** 있다
- "이 사람은 이 상황에서 이렇게 판단했다"
- "이 지식과 저 경험이 이렇게 연결되었다"
- 판단력, 맥락, 연결 — **텍스트에는 없는 것**

OU는 데이터 플랫폼이면서 동시에, **인간 사고의 구조를 디지털로 옮기는 프로젝트**다.
