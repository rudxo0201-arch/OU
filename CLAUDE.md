## 0. OU가 풀려는 문제

사람들은 일정, 할일, 습관, 알람, 일기, 노트, 자료, 가계부, 메신저, 사진, 음악, 운동 기록을 **각각 다른 앱**에 넣는다. 각 앱은 자기 데이터를 자기 안에서만 본다. 같은 사람이 같은 사건을 다섯 군데에 쪼개 입력하고 다섯 군데에서 따로 검색한다. **컨텍스트가 도구마다 끊긴다.**

LLM 시대에 이 단절은 더 어색해졌다. "지난주에 약속한 그 사람과의 미팅이 언제였지?"는 LLM이 그 데이터를 한꺼번에 보지 않는 한 풀 수 없다. 그래서 OU는 **"말하면 OU가 알아서 어디에 어떻게 저장할지 결정한다. 찾으면 OU가 알아서 어떤 뷰로 보여줄지 결정한다"** 라는 구조를 택한다. 사용자는 도구를 고르는 데 정신을 쓰지 않고, 자기 데이터 우주에 자연어로 말한다.단, 초반에는 한 데이터 인풋 장소에서 빠르게 도메인을 분류하고 자동으로 적절한 도구(orb, DB)에 안착되기 어려우므로 각 orb별 인풋창을 적극 활용한다. 각 인풋창에는 도메인 프롬프팅이 되어있고, orb 특화된 프롬프트가 작동한다.

이걸 OU 내부에서는 **개인 데이터 OS**라고 부른다. 외부에는 이 단어를 쓰지 않는다(이유는 §2 참조).

## 1. 차별점 — 왜 기존 사내 SaaS·챗봇이 이 자리를 못 채우는가

요즘 회사들은 사내 챗봇·내부 위키·자체 LLM 컨텍스트를 만들어 사내 업무를 체계화한다. 잘 만든 곳도 많다. 그러나 그 데이터는 **회사 안에서만 닫혀 있다.** 회사가 자기 직원의 행동은 본다. 자기 고객의 진짜 일상은 못 본다. 고객이 그 회사 앱을 켰을 때만, 켜는 동안만, 켜는 만큼만 본다. 결과적으로 회사와 고객 사이에는 늘 **데이터 단절**이 있다.

OU는 다르다. OU의 회원(C)은 자기 데이터 우주를 OU 위에 쌓는다. 회사(B)가 OU에 들어와 서비스를 운영하면, 회사도 같은 OU 위에 있다. **B와 C가 같은 데이터 기반을 공유한다.** OU 입장에서는 둘 다 똑같이 OU의 회원이다. 그래서:

- 회사는 자기 고객(C)의 행동·소비 패턴을 OU에서 분석할 수 있다 (회원이 권한을 주는 범위에서)
- 회사가 고객과 소통·마케팅하는 창구가 별도 채널이 아니라 **OU 한 곳**에 응집된다
- 회사가 새 기능을 출시할 때 자기 회원 DB 외에 OU의 데이터 표준을 그대로 활용할 수 있다

이걸 한 줄로 줄이면 "B/C 동격"이지만, 그 한 줄 표어만 보면 무엇을 푸는지 알 수 없다. 핵심은 **고객과 끊기지 않는 SaaS 통합 OS**라는 것이다. 이게 사내 챗봇·내부 SaaS 시장과 OU의 진짜 차이다.

이 비전은 **장기**다. Phase 1에서 회사를 끌어들이지 않는다. Phase 1은 일반 회원(C)의 데이터 우주를 먼저 채우는 단계다. C가 충분히 쌓여야 B를 끌어들일 명분이 생기기 때문이다.

C는 개인 사용자뿐 아니라 **회사(B)의 고객도 C**다. 회사 직원이 아니라 그 회사 서비스를 이용하는 일반 고객이 OU에 자기 데이터 우주를 쌓으면 그 고객도 C다. B와 C가 동격인 이유가 여기 있다 — OU 안에서 회사도, 그 회사의 고객도 모두 '회원'이라는 단일 레이어 위에 있다.

## 2. 외부 포지셔닝과 해금 구조

회원에게 "개인 데이터 OS"라고 소개하지 않는다. 보통 사람이 "OS"라는 단어로 직관을 갖기 어렵다. "내 데이터 우주를 만들자"는 추상은 가입 후 1주일 안에 가치를 증명하기 힘들다. 그래서 외부에 OU를 던질 때는 **"LLM을 활용한 일정 앱"** 으로 보여준다. 사용자가 받아들이기 쉬운 작은 약속(일정만 잘 관리해 줄게)으로 시작해서, 쓰는 동안 자연스럽게 다른 영역을 OU에 넘기게 만든다. 이걸 **해금 구조**라고 부른다.

해금 순서(대학생 일상을 따른다):

```
Phase 1   일정 + 알람        ← 우선 진입점
Phase 1   할일 + 습관        ← 일정 안에서 자연스럽게 등장
Phase 1   일기 / 한 일 기록  ← 일정과 같은 시간축 위
Phase 2+  노트 / 자료 / 과제 관리
Phase 2+  가계부, 운동, 사진, ...
```

이 순서를 일정→할일→습관→일기로 묶은 이유는 네 도메인 모두 **"시간축 위의 행위"** 라는 공통 골격을 가지기 때문이다. 데이터 필드(언제·무엇을·했는지/할 건지)와 뷰(달력·목록·체크박스)가 강하게 공유된다. 이 골격을 잘 만들면 노트·자료·과제 같은 **시간축이 약한 도메인**이 같은 패턴 위에 얹힌다. 반대로 이질적인 도메인(가계부의 금액축, 사진의 미디어축)을 먼저 시도하면 골격이 제대로 안 잡힌 채로 분기가 늘어나 코드가 흩어진다.

## 3. Phase 1 타겟과 도메인

**타겟**: 캘린더 앱, 할일 앱, 습관 앱, 알람 앱을 따로 쓰느라 지친 일반인. 우선 **대학생**.

대학생을 첫 타겟으로 고른 이유는 (1) 분산된 앱을 가장 많이 쓰고, (2) 새 앱을 시도하는 비용이 낮으며, (3) 일정→과제→노트→자료로 이어지는 학생 일상이 OU의 해금 구조와 잘 맞기 때문이다. 대학생이 OU에 머무는 시간이 길어지면 그 자체로 OU가 "공부·일상이 모이는 환경"이 된다.

**Phase 1 도메인 4개** (모두 Item의 한 종류):

| 도메인 | data 필드 핵심 | 대표 뷰 |
|--------|---------------|---------|
| `schedule` | 시작 시각, 종료 시각, 장소 | 캘린더, 컨투어, 목록 |
| `task` | 마감, 우선순위, 완료 여부 | 체크리스트, 칸반 |
| `habit` | 반복 주기, 연속 일수 | 트래커, 캘린더 도트 |
| `journal` | 기록 시각, 본문 | 타임라인, 일기 카드 |

이 네 도메인은 **모두 시간축 위에 있다**. data 스키마가 공유되는 부분이 많고, 입력 흐름(자연어 → 도메인 분류 → data 추출)도 같다. 그래서 네 도메인은 같은 코드 골격 위에서 *데이터*만 다르게 들어가도록 구현한다. 도메인별로 별도 코드 트리를 만들기 시작하면 곧 다섯 번째 도메인이 들어올 때마다 비슷한 코드를 또 쓴다.

## 4. 핵심 용어 — 왜 Item 한 단어로 단순화했는가

이전 OU 문서들은 `DataNode`, `Section`, `Sentence`, `Triple` 같은 계층 용어를 썼다. 의미는 깔끔하다. 메시지를 노드로 묶고, 노드를 섹션으로 자르고, 섹션을 문장으로 쪼개고, 문장에서 트리플을 추출하는 5계층. 그러나 이 추상은 Phase 1 가치 대비 비용이 너무 크다. 신규 도메인 추가, 디버깅, 뷰 매핑, 마이그레이션이 5계층 위에서 일어나면 작업마다 5계층의 정합성을 신경 써야 한다.

그래서 Phase 1에서는 모든 단위를 **Item** 하나로 부른다. Item 하나가 일정 1개, 할일 1개, 습관 1개, 일기 1개를 표현한다. 도메인은 Item의 필드(`domain: 'schedule'` 등)로 분기한다. 노드/섹션/문장은 백그라운드 분석 단계에서만 의미를 갖고, 사용자가 직접 보는 단위는 항상 Item이다. 이렇게 단순화하면 새 도메인을 추가할 때 *Item의 새 도메인 값과 그 도메인의 data 스키마*만 정의하면 되고, 뷰는 도메인을 입력으로 받는 컴포넌트로 통일된다.

> **⚠️ 용어 정의의 코드 원본**: 아래 표는 개념 설명용 요약이다. **실제 식별자 문자열(도메인값, 라우트, 위젯 타입, UI 레이블)의 원본은 `src/lib/ou-registry.ts`**다. 용어를 재정의하거나 추가할 때 이 표를 먼저 수정하지 말고 `ou-registry.ts`를 수정한다 — TypeScript 컴파일러가 모든 사용처를 자동으로 찾아준다.

| 용어 | 정의 | 코드 원본 | 안 쓰는 표현 |
|------|------|-----------|--------------|
| **Item** | 의미 단위로 구조화된 데이터 1개 (일정·할일·습관·일기 모두) | `TERMS.ITEM` | DataNode, 항목, 카드, 노드 |
| **도메인 (domain)** | Item의 종류 분류. Phase 1 = `schedule` / `task` / `habit` / `journal` | `DOMAINS.*` | (없음) |
| **Orb** | 여러 모듈을 조합한 미니앱. OU 백엔드 공유, 1+ 페이지 가능 | `TERMS.ORB`, `ORB_SLUGS.*` | 채팅창=Orb (X) |
| **모듈 (Module)** | 원자적 프론트엔드 컴포넌트 (Input / Display / Logic) | `TERMS.MODULE` | 위젯 |
| **뷰 (View)** | Item을 특정 시각화로 렌더링한 결과 | `TERMS.VIEW` | 데이터뷰(중복) |
| **회원** | OU에 데이터 우주를 쌓는 주체 (개인이든 회사든) | `TERMS.MEMBER` | 유저 |
| **관리자** | OU 운영팀 | — | (없음) |

**B와 C가 모두 회원**이라는 §1의 원칙이 이 표에 그대로 박혀 있다. 회사도 회원, 개인도 회원이다.

## 5. 데이터 모델 — DB 스키마 재설계 예정

DB 스키마(`supabase/migrations/`)는 messages → data_nodes → sections → sentences → triples 5계층을 가정하고 만들어졌다. Phase 1 가치 대비 이 추상은 비싸다 — 신규 도메인을 하나 더할 때마다 다섯 계층의 정합성을 함께 신경 써야 한다. 그러나 현 코드가 이 스키마를 깊게 참조하기 때문에 **즉시 바꾸지는 않는다**. Phase 2 코드 정리 시점에 Item 중심으로 평탄화한다.

지금 단계에서 지킬 원칙:

- **신규 테이블은 보수적으로**. 곧 평탄화될 가능성을 알면서 새 테이블을 만들면 마이그레이션 부채만 늘어난다.
- **원본 메시지·파일은 R2에 보존한다**. 파싱이 실패해도 원본은 남아 있어야 나중에 다시 시도할 수 있다. 데이터 손실은 OU에서 가장 비싼 종류의 실수다.
- **데이터는 삭제하지 않는다**. 모든 변경은 이벤트로 쌓는다. 이게 백그라운드 트리플·임베딩·검색을 가능하게 한다.
- **트리플(subject-predicate-object)은 백그라운드 자산이다**. 사용자에게 즉시 가치를 만들어주지 않으면 Phase 1에서는 추출을 미룬다. 단 추출 자체를 영구히 버리지는 않는다 — Phase 2 이후 의미 검색·관계 그래프의 토대가 된다.
- **`emotion` 같은 stealth 도메인은 조용히 기록한다**. 회원이 "내 감정 기록 어디 있어?"라고 명시적으로 묻기 전까지 뷰에 노출하지 않는다. 자기 감정을 스스로 추적당하고 있다고 느끼는 순간 신뢰가 깨지기 때문이다.

## 6. 뷰 모델 — 왜 LLM이 뷰를 매번 만들지 않는가

가능하면 LLM이 사용자 데이터를 보고 즉석에서 뷰를 생성하면 멋질 것 같다. 그러나 Phase 1에서 그렇게 가지 않는다. 이유는 셋이다. (1) **느리다** — 뷰 한 번 보려고 LLM 호출 몇 초 기다리면 사용성이 무너진다. (2) **비싸다** — 회원 한 명이 하루 수십 번 뷰를 토글하면 LLM 비용이 단가를 먹는다. (3) **일관성이 없다** — 같은 데이터를 보여줄 때 매번 다른 레이아웃이 나오면 회원은 "OU가 이상하다"라고 느낀다.

그래서 Phase 1의 뷰는 OU 팀이 **미리 정성껏 만든 카탈로그**에서 나온다. AI의 역할은 *생성*이 아니라 *선택*이다:

```
입력 → 도메인/타입 추출 → AI가 카탈로그에서 적합한 뷰 1개 선택 → 맵핑 + 필터 + 정렬 → 렌더
```

회원은 다른 뷰로 직접 토글할 수 있다. 토글은 Phase 1의 핵심 학습 데이터다 — "이 도메인의 이 데이터에서 회원들은 어떤 뷰를 더 자주 고르나"가 카탈로그를 다듬는 신호가 된다. Phase 2 이후 LLM이 새 뷰를 동적으로 만드는 로직은 그 학습 데이터 위에서 시작한다.

뷰 추가 = `views/registry.ts`에 등록만. 기존 뷰 코드를 건드리지 않는다. 뷰가 *선택자*이지 *생성자*가 아닌 이상, 카탈로그 확장이 곧 OU의 표현력 확장이다.

## 7. 입력 모델 — 모듈 카탈로그

지금 코드에는 비슷한 입력 UI가 5중 6중으로 만들어져 있다 — `QSBar`, `OrbChat`, `OrbInputBar`, `ChatInput`, `OUChatWindow`, `OrbFullscreen`, `OrbQuick`, `OrbSearch`, `OrbPost`, `OrbComment` 등. 각각 이유가 있어 만들어졌지만, 결과적으로 *같은 역할*을 하는 입력바가 여러 형태로 흩어져 있다. 작업할 때 어느 입력바를 고쳐야 하는지가 모호해서 새 기능이 들어올 때마다 또 한 벌이 생긴다.

Phase 1의 원칙: **입력은 명확한 역할을 가진 "모듈" 단위로 카탈로그화한다.** 같은 역할의 모듈을 두 벌 만들지 않는다. 코드에 현재 있는 `QSBar`는 그 카탈로그 안의 *한 모듈* — "QS LLM 모듈"로 본다. 음성 입력, 검색 입력, 댓글 입력 같은 다른 형태의 입력 모듈도 들어올 수 있고, 각각 독립된 책임을 가진다.

QS LLM 모듈의 최종 형태는 아직 확정이 아니다. 이 결정은 코드 정리 단계에서 사용성을 보면서 내린다. 지금은 "여러 입력 모듈이 있을 수 있고, 그중 하나가 QS LLM 모듈이다"라는 *구조 원칙*만 SSOT다.

**입력 모듈 역할 분리 (D-002)**:

- `/home` QSBar — **inbox 역할**. 도메인이 불명확한 빠른 입력을 받아 백그라운드로 DB화한다. 분류·파싱은 나중에 처리. `/api/quick`을 사용.
- `/orb/[slug]` 입력창 — **도메인 전용 역할**. 도메인이 이미 확정된 입력창이므로 해당 도메인 전용 프롬프트(extraction/domains/{domain}.ts)를 사용해 LLM이 동기 파싱한다. 모호한 경우 clarification 질문을 반환하고 저장을 보류한다. `/api/orb-input`을 사용.

두 역할을 같은 API로 처리하면 도메인 전용 프롬프트가 무시되고 즉시 저장으로 모호 판단 기회가 사라진다. 이 구분이 §7의 핵심 설계 원칙이다.

## 8. 라우트 — 왜 3개로 좁혔는가

Phase 1 사용자 라우트는 **3개**다:

| 라우트 | 역할 |
|--------|------|
| `/` | 랜딩 — 비로그인 진입점. 외부 포지셔닝(LLM 일정 앱)을 보여준다 |
| `/home` | 로그인 후 메인 — Item 대시보드, 뷰가 떠 있는 곳 |
| `/orb/[slug]` | 개별 Orb 전체 화면 |

라우트 그룹은 `(auth)` / `(main)` / `(public)` 셋으로 정리되어 있고, 같은 기능을 두 그룹에 두지 않는다. 새 기능을 만들 때 새 라우트를 미리 만들지 않는다 — 셋 중 하나에 속하는지 결정한 다음 그 안에 둔다. **3개로 좁힌 이유는 단순함이 아니라 *결정 가능성***이다. 새 기능이 들어올 때 어디에 둘지가 즉시 정해지지 않으면, 비슷한 페이지가 두 그룹에 동시에 살아 있는 상태로 표류한다 — 그게 정확히 과거 OU에서 일어난 일이다.

## 9. UI / 디자인 — 왜 UI 라이브러리를 쓰지 않는가

OU는 만들 UI의 폭이 큰 제품이다. 입력 모듈 카탈로그, 뷰 카탈로그, Orb 조합, 도메인별 Display, 그래프 시각화, 시간축 컴포넌트, 실시간 알림 등이 모두 하나의 시각 언어 안에서 일관되게 움직여야 한다. Mantine·MUI·Chakra 같은 UI 라이브러리는 **기본 컴포넌트의 추상화 가정**(스타일 시스템, props 모양, 내부 상태 모델)이 강하다. OU가 만드는 *비표준 컴포넌트* — 시간축 위에서 드래그되는 Item 카드, 동적으로 바뀌는 뷰 컨테이너, 모듈을 임베드한 Orb — 는 그 가정과 자주 충돌한다. 라이브러리를 우회하거나 덮어쓰는 코드를 매번 짜는 순간 라이브러리의 이점이 사라진다.

그래서 OU는 **순수 CSS + 토큰 시스템** 위에 자체 디자인 시스템을 만든다. 컴포넌트 자유도가 높아지고, 디자인 토큰을 한 곳에서 통제할 수 있고, 새 비표준 시각 패턴을 만들 때 라이브러리와 싸우지 않는다.

DS 컴포넌트는 `Ou*` 접두사로 통일한다. 잔여 `Neu*` 컴포넌트는 점진적으로 `Ou*`로 통합한다(§14). 새 컴포넌트는 모두 `Ou*`로 만든다.

추가 원칙:

- **이모지를 아이콘으로 쓰지 않는다** — `icon={"🏠"}` 같은 이모지 아이콘은 절대 금지. OS마다 렌더가 다르고, 크기·색상 제어가 불가능하다. 아이콘이 필요하면 인라인 SVG 또는 SVG 컴포넌트를 쓴다. 이 규칙은 `pnpm build` 하네스(`scripts/check-orb.mjs`)가 자동 검출한다.
- **로고는 `OuLogo` 컴포넌트(SVG) 사용** — 텍스트로 "OU"나 로고를 렌더링하지 않는다. 어디에 로고가 필요하든 `import { OuLogo } from '@/components/ds'`를 쓴다. 이 규칙을 어기면 브랜드가 깨진다.
- **강조·타이틀 폰트 = `var(--ou-font-display)` = Orbitron** — 큰 제목, 숫자 강조, 섹션 타이틀에 `var(--ou-font-display)`를 쓴다. DM Sans나 다른 폰트를 타이틀에 직접 지정하지 않는다. `var(--ou-font-body)`는 본문 전용. 이 두 토큰이 폰트 일관성의 전부다.
- **데스크톱 퍼스트** — OU의 정보 밀도(여러 도메인이 동시에 보이는 대시보드, 그래프뷰)는 데스크톱에서 먼저 자리 잡는다. 모바일은 데스크톱이 자리 잡은 후 좁힌다. `@media` breakpoint 768/1024/1440 사용. 모바일 단일 컬럼을 강제하지 않는다.
- **인증 게이트** — 비로그인 회원이 클릭한 버튼을 `disabled`로 막지 않는다. 클릭하면 `/login`으로 보낸다. 막힌 버튼은 *"무엇을 할 수 있는지"* 를 회원에게 못 보여주기 때문이다.
- **DS 신규 컴포넌트** — 새 UI 컴포넌트가 필요하면 `src/components/ds/GOVERNANCE.md` 절차에 따라 RFC 제출 후 인준을 받고, `src/components/ds/registry.ts`의 `DS_COMPONENTS`에 메타데이터를 추가한다. `registry.ts`가 DS 컴포넌트의 SSOT다 — GOVERNANCE.md는 절차서, `registry.ts`는 정의 원본. DS 변경(추가·deprecated 마킹 등)이 발생하면 DECISIONS.json에 `type: 'ds'`로 등록한다. DS를 우회한 inline `<button style={{...}}>` 또는 카드형 `<div>` 작성 금지.

## 10. 기술 스택 — 왜 그대로 두는가

현재 스택은 OU의 핵심 요구(데이터·LLM·실시간·인증·검색)를 모두 푼다. 바꾸지 않는 결정도 결정이다.

| 영역 | 기술 | 그대로 두는 이유 |
|------|------|------------------|
| Framework | Next.js 14 (App Router) + TypeScript | App Router로 라우트 그룹·서버 컴포넌트·스트리밍이 자연스럽다 |
| State | Zustand | 가벼움. 추상화 비용 없이 OU의 모듈 단위 상태에 맞는다 |
| 인증 | Supabase Auth | Supabase DB와 한 번에 통합 |
| SQL DB | Supabase PostgreSQL | OU 데이터 모델에 충분, 성장 시 Neon으로 이전 |
| 벡터 DB | pgvector | 위와 같은 인스턴스에서 처리 |
| LLM 채팅 | Claude Sonnet (스트리밍) | OU의 자연어 입력→구조화에 가장 결과 일관성 |
| LLM 배치 | Claude Haiku | 빠르고 싸고, 도메인 분류에 충분 |
| LLM Fallback | OpenAI | Claude 장애 대비 |
| 임베딩 | text-embedding-3-small | 비용·성능 균형 |
| OCR | Gemini Vision | 한국어 OCR이 OpenAI보다 안정 |
| 그래프 | PixiJS + d3-force Web Worker | 60fps 그래프뷰 요구 |
| 파일 스토리지 | Cloudflare R2 | 원본 보존 비용이 가장 싸다 |
| 캐싱 | Upstash Redis | 서버리스 친화 |
| 실시간 | Supabase Realtime | 인증·DB와 한 인스턴스 |
| 배포 | Vercel / ouuniverse.com | Next.js 기본값 |

스택 변경이 필요해지는 순간은 (1) 비용이 회원당 단가를 깬다, (2) 성능이 OU의 핵심 UX(빠르게, 원하는 대로) 를 못 받친다, 둘 중 하나다. 그 전까지는 그대로 둔다.

## 11. Claude Code가 OU 코드를 만질 때

Claude Code가 사용자에게 "수정 완료"라고 보고하는 순간, 사용자는 그 코드가 빌드되고 동작한다고 가정한다. 빌드 실패한 코드가 그대로 누적되면 다음 작업이 깨진 상태 위에 쌓인다. 그래서 작업이 끝났다고 판단하기 전 두 단계를 거친다:

1. `pnpm build` — 타입 에러 없음, Orb harness 통과 확인
2. `pnpm test:e2e` (또는 `/verify` 슬래시 명령) — 관련 spec 실행, 런타임 동선 + 디자인 CSS 검증

"빌드 통과"는 컴파일 정합만 보장한다. 라우터 push 경로 오타, 클릭 핸들러 누락, CSS 누락은 빌드를 통과한다. `pnpm test:e2e`가 진짜 "동작한다"의 보증이다. spec 없는 영역은 "이 변경에 대한 spec 없음. 사용자 시각 확인 필요"라고 명시한다 — 검증된 척 하지 않는다.

스크린샷 baseline diff 실패 시 자동 갱신하지 않는다. 반드시 "의도된 변경인가요?"를 사용자에게 확인 후 `pnpm test:e2e:update`로 갱신한다.

`pnpm dev`가 돌고 있는 동안 `pnpm build`나 `rm -rf .next`를 하지 않는다. dev와 build가 같은 `.next` 디렉토리를 동시에 쓰면 캐시 파일이 충돌해서 dev 서버가 알 수 없는 상태가 된다. 과거에 이로 인해 사용자가 `rm -rf .next && pnpm dev`를 반복하는 일이 잦았고, 그 반복 자체가 별도 문제를 만들었다. dev가 실행 중이면 dev를 끄거나 새 터미널에서 작업하고, dev에서 직접 빌드 검증을 한다.

한 번에 1개 파일만 고친다. 작은 단위 확인은 시간 낭비처럼 보이지만, 한 번에 여러 파일을 고치면 어느 변경이 무엇을 깼는지 추적이 안 된다. 디버깅 비용을 가장 크게 줄이는 단일 습관이다.

에러를 만나면 추측으로 고치지 않는다. 로그·콘솔 메시지를 먼저 본다. 추측으로 고치면 진짜 원인이 다른 곳에 남아 다시 터지고, 가짜 수정으로 표면만 덮으면 다음 작업이 그 위에 또 쌓인다.

API 키는 서버사이드에서만 다룬다. 클라이언트에 노출되는 순간 즉시 LLM/저장소 비용이 폭발할 위험이 있어서 이 규칙만큼은 협상하지 않는다.

**파싱은 LLM 단독으로 한다. 정규식을 쓰지 않는다 (D-001).** 초반에 데이터가 적을 때 정규식은 서비스 품질 저하와 모호 처리 불가 문제를 일으킨다. "12기 점심"처럼 해석이 두 가지 이상인 입력을 정규식은 임의로 결정하지만 LLM은 사용자에게 되물을 수 있다. 비용 최적화는 데이터가 충분히 쌓인 뒤 논의한다. `/api/orb-input`(도메인 전용)·`/api/quick`(QSBar inbox) 모두 LLM 단독으로 동작한다.

세션 시작 시 `DECISIONS.json`을 읽어 진행 중인 작업과 대기 중인 결정을 파악한다. 코드에 정규식이 있어도 "이유가 있겠지"로 따라가지 않는다. 문서가 코드보다 권위가 있다.

신규 UI 컴포넌트에는 Mantine 등 UI 라이브러리를 import하지 않는다(§9). OU의 자체 DS와 순수 CSS 토큰을 쓴다.

새 라우트를 미리 만들지 않는다(§8). 새 기능이 들어오면 `/`, `/home`, `/orb/[slug]` 셋 중 어디에 속하는지 결정한 다음 그 안에 둔다. 새 라우트가 정말 필요해지면 그때 SSOT를 고친다.

위젯 타입(`registerWidget`의 `type` 필드)은 Supabase에 저장되는 **영구 식별자**다. 타입명을 변경하거나 통합할 때, 구 타입 등록을 지우기 전에 반드시 `src/components/widgets/registry.ts`의 `WIDGET_TYPE_MIGRATIONS`에 `{ 구타입: '신타입' }` 항목을 추가한다. 이렇게 하면 DB에 저장된 구 타입이 렌더 시 자동으로 신 타입으로 리매핑된다. 타입을 코드에서만 지우면 사용자 DB에 저장된 위젯이 조용히 사라지기 때문이다 — 2026-04-26 `'qsbar'` 제거 사고에서 확인됨.

## 12. MCP / 메모리 정책

- `mcp__ou__record_message`: **일단 비활성**. DB 스키마 재설계가 진행 중인 동안 자동 기록을 켜두면 옛 스키마로 데이터가 계속 쌓여서 마이그레이션 비용이 늘어난다. 안정화 후 재개.
- `mcp__ou__search_universe`: 사용 가능하지만 **결과는 참조용**이다. 옛 데이터에는 옛 컨셉(DataNode, /my 라우트 등)이 박혀 있을 수 있어서 그 표현을 그대로 신 코드에 옮기지 않는다.
- `~/.claude/.../memory/`: 2026-04-25에 OU 프로젝트 메모리 49개 전체 박제. 새 결정·취향이 쌓이면 새 메모리로 추가한다(구 49개는 `_archive/2026-04-25/` 참조 전용).
